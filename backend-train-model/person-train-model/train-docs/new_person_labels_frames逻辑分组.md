## 1. 本文档解决什么问题

本文档专门说明：

1. `new_person_labels` 当前这批图片，自动化分组到底依据什么；
2. 为什么当前选择 `feature_threshold=0.06` 这版结果作为主参考；
3. 自动粗分组之后，人工还需要怎么继续细分；

5. 已经创建好的物理分组目录结构在哪里、后续该怎么用。

## 2. 当前采用哪一版自动分组结果作为主参考

当前已经试过三档自动粗分组阈值：

- `0.08`
- `0.06`
- `0.04`

当前推荐：

> **优先以 `feature_threshold=0.06` 这版结果作为“人工继续细分前的主参考版本”。**

原因如下：

### 2.1 `0.08` 的问题：偏粗

- 总组数只有 `5` 组；
- 其中存在很大的组：`1139` 张、`582` 张；
- 说明虽然已经比“整包 flat source 不分组”好，但仍然容易把多个场景或视角混在同一组。

### 2.2 `0.04` 的问题：偏细

- 总组数上升到 `9` 组；
- 已经开始出现很小的组：`41` 张、`56` 张、`72` 张；
- 这类结果更适合拿来检查“有没有明显异类小场景”，但不适合作为当前第一版正式执行口径。

### 2.3 `0.06` 的优点：最平衡

- 总组数：`6` 组；
- `1280x720` 保持为 `1` 组，共 `272` 张；
- `1920x1080` 被拆成 `5` 组，共 `2235` 张；
- 相比 `0.08`，已经把过大的混合组继续拆开；
- 相比 `0.04`，又没有碎到后续管理成本明显上升。

因此当前更稳妥的执行方式是：

> **先拿 `0.06` 这版结果做人工复核和物理分组，再根据代表帧情况决定哪些组继续细拆。**

## 3. 自动化分组的依据是什么

这一部分尽量写细，避免后续只记住“脚本给了几个组”，却不知道为什么会这样分。

### 3.1 第一步：先按图片分辨率分桶

脚本首先按：

- 图片宽度 `width`
- 图片高度 `height`

进行第一层分桶。

### 中文注解

这一步的含义是：

> **先把明显不属于同一成像规格的图片拆开。**

如果两张图分辨率不同，例如：

- `1280 x 720`
- `1920 x 1080`

那么它们大概率至少在下面某一项上不同：

- 摄像头来源不同；
- 视频导出规格不同；
- 画面裁切方式不同；
- 后续 ROI 边界比例关系不同。

所以分辨率分桶不是为了“好看”，而是为了先做第一层物理隔离。

### 3.2 第二步：对同一分辨率桶内的图片提取粗视觉特征

当前脚本使用了两类特征：

1. **灰度缩略图特征**
   - 把图片转为灰度；
   - 做轻微模糊；
   - 缩放到 `24 x 24`；
   - 展平成一个低维布局向量。
2. **边缘缩略图特征**
   - 对灰度图做 `Canny` 边缘检测；
   - 同样缩放到 `24 x 24`；
   - 展平成边缘结构向量。

最终把这两部分拼接起来，形成每张图片的粗视觉特征。

### 中文注解

这一步的目的不是识别“图片里具体是谁”，而是抓住：

- 大致背景布局；
- 地面 / 设备 / 墙体的轮廓结构；
- 视角是不是差不多；
- 主要边界线位置是不是接近。

也就是说，脚本更关注：

> **这些图片看起来是不是来自同一种场景构图。**

### 3.3 第三步：按特征距离做聚类分组

在同一分辨率桶内，脚本会逐图比较：

- 当前图片特征
- 当前已有组的中心特征

使用的距离是：

> **平均绝对差（mean absolute difference）**

如果距离小于当前阈值，就并入已有组；
否则新开一组。

### 中文注解

这一步可以理解成：

> **如果一张图和某一组“整体看起来足够像”，就放进这一组；如果不像，就单独分出去。**

阈值越小，分组越细；
阈值越大，分组越粗。

## 4. 当前 `0.06` 结果的详细说明

### 4.1 当前统计结果

当前 `feature_threshold=0.06` 的统计结果是：

- 总图片数：`2507`
- 总组数：`6`

按分辨率拆开后：

- `1280x720`：`1` 组，共 `272` 张
- `1920x1080`：`5` 组，共 `2235` 张

6 个组的图片数分别是：

- `group_0001`：`272`
- `group_0002`：`449`
- `group_0003`：`127`
- `group_0004`：`387`
- `group_0005`：`582`
- `group_0006`：`690`

### 4.2 这些结果该怎么理解

这说明：

1. `new_person_labels` **不是一个统一稳定场景源**；
2. 至少从粗视觉布局看，它已经能被拆成多个候选场景组；
3. 但其中仍然存在大组，说明自动分组还没有替代人工收口。

### 4.3 当前组的优先级建议

从人工继续处理角度，建议这样看：

#### 更适合优先检查、并有机会直接进入组级 ROI 的组

- `group_0001`：`272`
- `group_0003`：`127`
- `group_0004`：`387`

原因：

- 规模适中；
- 更像可能已经接近单一稳定场景；
- 后续人工检查成本较低。

#### 优先需要继续人工细拆的大组

- `group_0002`：`449`
- `group_0005`：`582`
- `group_0006`：`690`

原因：

- 组仍偏大；
- 更可能还混有多个视角 / 场景；
- 不建议直接拿去统一画组级 ROI。

## 5. 自动化分组后的最终产物路径

### 5.1 自动粗分组主结果路径

当前用于正式参考的自动粗分组结果路径是：

- 汇总 JSON：
  - `backend-train-model/person-train-model/train-result/working/new_person_labels_roi_grouping_t006/group_summary.json`
- 逐图 manifest：
  - `backend-train-model/person-train-model/train-result/working/new_person_labels_roi_grouping_t006/group_manifest.csv`

这两个文件的作用分别是：

- `group_summary.json`：看每组大小、代表帧；
- `group_manifest.csv`：看每张图当前被分到了哪一组。

### 5.2 已创建的物理分组目录结构路径

当前已经直接创建好的**物理分组目录模板**路径是：

- `backend-train-model/person-train-model/train-result/working/new_person_labels_frames_grouping_t006/grouped/`

当前已创建的分组子目录包括：

```text
grouped/
├─ group_0001/
│  ├─ images/
│  └─ labels/
├─ group_0002/
│  ├─ images/
│  └─ labels/
├─ group_0003/
│  ├─ images/
│  └─ labels/
├─ group_0004/
│  ├─ images/
│  └─ labels/
├─ group_0005/
│  ├─ images/
│  └─ labels/

   ├─ images/
```

### 中文注解

这套 `grouped/` 目录当前应理解为：

> **后续人工物理分组和 ROI 标注的工作目录模板。**

它不是说：

- 当前 fullframe 主线已经切到这里训练；
- 当前这些目录里的图片一定已经是最终逻辑组。

它只是：

- 为后续人工继续细分、重命名、放图、放标签提供一个稳定起点。

## 6. 人工进行分组，具体需要怎么做

这一部分是最重要的操作说明。

### 6.1 第一步：先看 `group_summary.json`，不要先盲搬文件

先做的不是马上复制图片，而是：

1. 打开 `group_summary.json`；
2. 看每个组的 `representatives`；
3. 先判断该组代表帧是否看起来来自同一种场景构图。

当前建议优先检查顺序：

1. `group_0002`
2. `group_0005`
3. `group_0006`
4. 再看 `group_0001 / group_0003 / group_0004`

原因是前 3 个更大，更可能需要继续细拆。

### 6.2 第二步：按“能不能共用 ROI”继续做人工判断

人工判断时，优先回答下面 5 个问题：

1. 是不是同一摄像头？
2. 是不是同一视角？
3. 背景和作业区边界是不是基本一致？
4. 图像裁切方式是不是基本一致？
5. 如果共用一版 ROI，大多数图片会不会都还能接受？

### 中文注解

这一步的核心不是问：

- “这些图是不是差不多”；

而是问：

> **“这些图是不是可以共用一版 ROI，而不会把 ROI 语义搞乱。”**

### 6.3 第三步：把组分成 A / B / C 三类

#### A 类：稳定组

满足：

- 同摄像头；
- 同视角；
- 背景和作业区边界稳定；
- 大多数图理论上可以共用一个 ROI。

动作：

> 进入组级 ROI 流程。

#### B 类：半稳定组

满足：

- 大体同场景；
- 但仍有轻微裁切变化、抖动、光照差异。

动作：

> 继续细分，再决定是否组级 ROI。

#### C 类：不稳定组

满足：

- 视角差别很大；
- ROI 边界变化明显；
- 混了多个场景；
- 不适合继续共用一个 ROI。

动作：

> 少量逐图 ROI，或者暂缓纳入 ROI-aware 第一版。

### 6.4 第四步：对大组做人工细拆

如果一个自动组太大，建议继续按下面维度拆：

- 白天 / 夜晚；
- 左偏 / 右偏；
- 远景 / 近景；
- 同一场景下不同构图版本；
- 同一分辨率但明显不同背景布局。

例如可以拆成：

- `group_0002_a_day`
- `group_0002_b_night`
- `group_0006_a_far`
- `group_0006_b_close`

### 6.5 第五步：再往物理目录里放图和标签

当你已经确认某一组或某一子组成立后，再把对应图片和标签放到：

- `grouped/<group_name>/images/`
- `grouped/<group_name>/labels/`

这一步可以：

- 复制；
- 硬链接；
- 或后续脚本化生成。

当前建议：

> **先完成逻辑判断，再正式往物理目录里落图。**

否则你会反复搬目录。

### 6.6 第六步：稳定组再进入 ROI 标注

只有当一组已经被你确认成 A 类稳定组后，才建议：

1. 给这组抽 `5~10` 张代表帧；
2. 用 Labelme 画组级 ROI；
3. 再做 `10~20` 张组内抽检；
4. 确认这组可以共用这版 ROI。

## 7. 当前最推荐的实际操作顺序

当前建议你按下面顺序推进：

```text
先看 t006 的 group_summary.json
-> 优先人工复核大组 group_0002 / group_0005 / group_0006
-> 把明显稳定的组先定成 A 类
-> 把大组继续细拆成子组
-> 再把确认后的组落到 grouped/ 目录
-> 最后只对稳定组先做组级 ROI
```

### 当前不建议这样做

1. 不建议直接拿 `0.08` 那版结果开画；
2. 不建议直接拿 `0.04` 那版结果当最终组；
3. 不建议在还没看代表帧之前，就把 `2507` 张图完全按自动结果当最终物理组；
4. 不建议默认所有组都统一做组级 ROI；
5. 不建议默认所有图都改成逐图 ROI。

---

## 8. 一句话执行口径

> **当前 `new_person_labels` 的 frames 逻辑分组，应优先以 `feature_threshold=0.06` 这版自动粗分组结果作为主参考：先按分辨率和粗视觉特征形成第一轮候选组，再通过代表帧人工判断哪些组可以共用 ROI、哪些组还要继续细拆，最后把确认后的稳定组落到 `new_person_labels_frames_grouping_t006/grouped/` 目录中，作为后续 ROI 标注和 ROI-aware 数据准备的正式工作目录。**
