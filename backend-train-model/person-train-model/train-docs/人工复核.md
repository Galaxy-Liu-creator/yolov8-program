## 2. 当前人工复核要解决什么问题

当前自动分桶已经完成，下一步人工复核的目标不是重新跑模型，而是把 hardest FN 继续拆成更贴近业务语义的原因，例如：

- 暗光 / 背光
- 遮挡
- 半身 / 截断
- 姿态异常
- 远景内部小人
- 边界远景小人
- 拥挤 / overlap
- 可疑标注问题

因此人工复核的核心产出是：

1. 每个代表帧 / 每个 FN GT 的人工语义标签；
2. 是否怀疑原始标注需要修框或重标；
3. 哪一类问题最值得进入下一轮补样或增强策略。

到当前 `5.13` 阶段，这里的重点需要再往前推进一步：

- 当前不再只是继续泛化补“语义桶”；
- 当前也不是看到 hardest FN 就立刻往训练参数上跳；
- 当前更关键的是：先把 hardest FN 正式拆成两条主线，再把 crowded / overlap 这条主线继续做成**机制级结论**。

也就是说，当前人工复核除了判断“这是拥挤 / overlap”，还要尽量继续回答：

1. 是不是一个框吃掉两个人；
2. 第二个人是不是几乎没有有效响应；
3. 还是其实有弱框，但 IoU / 匹配不过线；
4. 又或者同时伴随明确标注问题。

4. 是不是只露出半身？

## 10. 人工语义标签到底怎么打

## 10.1 当前建议使用的语义标签

建议优先使用这 9 个：

1. `dark_or_backlit`
2. `occluded`
3. `partial_body`
4. `pose_or_shape_unusual`
5. `small_far_interior`
6. `small_far_boundary`
7. `crowded_or_overlap`
8. `annotation_or_label_suspect`
9. `mixed_hard_case`

## 10.2 各标签怎么理解

### `dark_or_backlit`

适用：

- 人体与背景对比度低；
- 背光导致轮廓不清；
- 不是特别小，但视觉上不容易稳定识别。

### `occluded`

适用：

- 被设备、立柱、车辆、其他人挡住一部分；
- 目标可见性明显下降。

### `partial_body`

适用：

- 只露出上半身；
- 下半身出画；
- 被画幅截断；
- 人体完整性明显不足。

### `pose_or_shape_unusual`

适用：

- 弯腰；
- 蹲姿；
- 转身；
- 肢体形态不像标准直立 person。

### `small_far_interior`

适用：

- 小；
- 远景；
- 位于画面内部；
- 不是贴边导致的问题。

### `small_far_boundary`

适用：

- 小；
- 远景；
- 明显贴边或贴角。

### `crowded_or_overlap`

适用：

- 多人近邻；
- 两个框相互干扰；
- 人和人挤在一起；
- 很像定位 / 匹配不过线问题。

### `annotation_or_label_suspect`

适用：

- GT 框范围明显不合理；
- 是否该标本身就可疑；
- 很像需要回到原始标注再确认。

### `mixed_hard_case`

适用：

- 同时符合 2~3 类；
- 很难只归因到一个主因；
- 例如：`远景内部小人 + 背光 + 半身`。

## 10.3 主标签和次标签怎么写

建议规则：

- `semantic_primary`：只写一个最主要原因；
- `semantic_secondary`：如果确实存在第二个明显原因，再写；
- 如果你一上来就觉得“什么都占一点”，不要立刻全写 `mixed_hard_case`，先尽量找主因。

例如：

- 主因：`small_far_interior`
- 次因：`dark_or_backlit`

或者：

- 主因：`occluded`
- 次因：`partial_body`

---

## 11. 什么情况下要判定 `need_relabel = true`

建议只有下面几种情况才标 `true`：

1. 原始框明显偏移；
2. 框明显过大或过小；
3. 明显漏标 person；
4. GT 本身是否该标都可疑；
5. 同一类样本在多个帧里都出现类似标注异常。

## 11.1 `relabel_status` 怎么写

- `not_needed`
  - 不需要修标。
- `suspected`
  - 怀疑有问题，但还没真正改。
- `fixed`
  - 已在临时工作目录修完。
- `pending`
  - 确认要改，但还没动手。

## 12. 第一轮人工复核完成后，需要更新哪些文件

第一轮看完后，相关记录主要落在这三个地方：

1. `stage_reviews/当前激活阶段/sequence_notes/sequence_*/5-xnotes.md`
   - 写序列级观察结论。
2. `stage_reviews/当前激活阶段/semantic_bucket_manifest.json`
   - 写逐 GT 结构化记录。
3. `stage_reviews/当前激活阶段/semantic_bucket_summary.md`
   - 写全局可读总结。

如果是**单人**复核，建议顺序：

> 先写当前阶段的 `5-xnotes.md` 和 `manifest.json`，最后再写当前阶段的 `summary.md`。

如果是**多人分工**，推荐顺序改成：

> 组员先写各自负责序列的当前阶段 `5-xnotes.md` ->
> 再由汇总人批量合并当前阶段 `semantic_bucket_manifest.json` ->
> 最后由组长或统一汇总人更新当前阶段 `semantic_bucket_summary.md`。

这样做的原因是：

- `notes.md` 是按序列拆开的，适合多人并行；
- 当前阶段 `semantic_bucket_manifest.json` 是单个 JSON 文件，多人同时改最容易冲突；
- 当前阶段 `semantic_bucket_summary.md` 是汇总结论，不适合每看完一张图就实时改。

另外：

- `by_source/review_asset_manifest.json` 是这轮素材导出清单；
- 它只负责说明“导出了哪些图、标签和 overlay”；
- **人工复核过程中不需要改这个文件。**

## 13. 组员统一填写规范

## 13.1 组员各自应该改哪些文件

推荐按下面口径执行：

### 13.1.1 可以直接发给组员的最简规则

如果你要把任务直接分给组员，最简单、最不容易出错的规则就是：

#### 组员默认只需要做两件事

1. 在 `by_source/` 里看自己负责的素材
2. 在当前激活阶段目录对应的 `sequence_notes/sequence_*/5-xnotes.md` 里写观察结论

也就是说，**组员默认主要修改的是各自负责序列的当前阶段 `5-xnotes.md`**。

#### 组员默认不要改的文件

除非你额外指定了专人负责汇总，否则组员**不要直接改**下面这些文件：

- `semantic_bucket_manifest.json`
- `semantic_bucket_summary.md`
- `by_source/review_asset_manifest.json`

以及：

- 顶层 `sequence_*/notes.md`
- 顶层 `semantic_bucket_manifest.json`
- 顶层 `semantic_bucket_summary.md`

原因是：

- 当前阶段 `semantic_bucket_manifest.json` 是单轮共用的单个 JSON，很多人同时改最容易冲突；
- 当前阶段 `semantic_bucket_summary.md` 是该轮最终总结文件，不适合每个组员边看边改；
- `by_source/review_asset_manifest.json` 只是素材导出索引，不是人工结论记录表。
- 顶层 `sequence_*/notes.md`、顶层 summary、顶层 manifest 现在都只是入口 / 兼容层，不再承载新阶段正文。

#### 谁负责改哪些文件

建议明确分工如下：

- **普通组员**：只改自己负责序列的当前阶段 `5-xnotes.md`
- **汇总人**：统一整理并更新当前阶段 `semantic_bucket_manifest.json`
- **组长 / 最终汇总人**：统一更新当前阶段 `semantic_bucket_summary.md`
- **素材维护人**：只有当重新导出或扩充素材时，才处理 `by_source/review_asset_manifest.json`

如果你不想让组员自己碰 JSON，那么可以要求他们：

> 只提交 `notes.md` 内容或按照统一表格把结果发给汇总人，
> 然后由汇总人统一更新 `semantic_bucket_manifest.json`。

### 组员日常复核时

优先修改：

1. 自己负责序列的 `sequence_*/notes.md`
2. 交给汇总人的结构化记录内容

不建议每个组员每看完一张图就直接去改：

- `semantic_bucket_summary.md`

因为这个文件是全局总结，应该按批次统一更新。

### 汇总阶段

由指定汇总人统一更新：

1. `semantic_bucket_manifest.json`
2. `semantic_bucket_summary.md`

### 明确不需要改的文件

- `by_source/review_asset_manifest.json`

这个文件只是素材导出索引，不是人工结论记录表。

### 13.1.2 推荐给组员的明确口径

为了避免误解，可以直接把下面这段话发给组员：

## 13.2 每条结构化记录怎么填

往 `semantic_bucket_manifest.json` 写一条记录时，建议按下面规则：

### `image_stem`

- 直接写图片 stem，不带扩展名；
- 必须与素材文件名一致。

示例：

- `D15_20260119061405_frame_0345`

### `sequence_name`

- 写该图片所属序列；
- 必须与 `notes.md` 所在序列目录一致。

示例：

- `D15_20260119061405`

### `split`

- 只允许写：`val` 或 `test`
- 可从 overlay 文件名对应的 run 推断，或从现有复盘结果中确认。

### `gt_index`

- 写该图片里这个 FN GT 的索引；
- 如果当前只先做帧级判断、还没精确到 GT 索引，不要乱填，先留待汇总人补。

### `algorithm_bucket`

- 只写当前 4 主桶之一：
  - `small_boundary_person`
  - `small_interior_person`
  - `medium_large_pose_or_appearance`
  - `crowded_or_localization`

### `semantic_primary`

- 只写一个主语义标签；
- 当前优先从下面这组里选：
  - `dark_or_backlit`
  - `occluded`
  - `partial_body`
  - `pose_or_shape_unusual`
  - `small_far_interior`
  - `small_far_boundary`
  - `crowded_or_overlap`
  - `annotation_or_label_suspect`
  - `mixed_hard_case`

### `semantic_secondary`

- 可空；
- 只有确实存在第二个明显原因时再写；
- 没有就写空字符串 `""`。

### `need_relabel`

- 只写布尔值：`true` / `false`
- 含义：这条样本是否怀疑需要修框、补标或回查原标注。

### `relabel_status`

- 只允许写：
  - `not_needed`
  - `suspected`
  - `fixed`
  - `pending`

推荐规则：

- 不需要改 -> `not_needed`
- 怀疑有问题但还没动 -> `suspected`
- 已在临时副本修完 -> `fixed`
- 确认该改但还没改 -> `pending`

### `notes`

- 用中文短句写肉眼依据；
- 不要只写“难”“小”“看不清”。

推荐写法：

- `远景内部小人，人体对比度低，偏背光`
- `半身 + 遮挡，下半身被设备挡住`
- `边界小人，但本质仍像素不足`
- `GT 框偏松，怀疑需要修框`

### `reviewed_by`

- 写复核人姓名、缩写或工号；
- 组内保持统一格式即可。

### `reviewed_at`

- 统一写 `YYYY-MM-DD`

示例：

- `2026-05-06`

## 13.3 `notes.md` 建议怎么写

`notes.md` 不是结构化表，它更适合写：

1. 这条序列主问题是什么；
2. 哪几张帧最有代表性；
3. 为什么判断成某类语义问题；
4. 是否怀疑存在标注问题；
5. 下一步建议动作。

建议每次至少写到这 3 类信息：

- **观察对象**：看了哪些帧
- **判断结论**：更偏哪类语义问题
- **是否需要修标**：有 / 无 / 待确认

## 13.4 推荐的团队协作节奏

如果你把任务分给多位组员，建议这样分工：

1. **组员 A/B/C**
   - 直接在 `by_source/` 里看图；
   - 各自负责某几个来源或序列；
   - 把结论先写到自己负责的 `sequence_*/notes.md`。

2. **汇总人**
   - 定期把组员确认过的结论合并到 `semantic_bucket_manifest.json`；
   - 检查字段拼写是否统一。

3. **组长 / 最终汇总人**
   - 在一批样本看完后，再更新 `semantic_bucket_summary.md`；
   - 输出阶段性结论和下一步建议。

这样最稳，因为：

- 多人同时改一个 JSON 容易冲突；
- 多人同时改 summary 也容易相互覆盖；
- `notes.md` 按序列拆开，更适合并行。

## 14. 当前最容易犯的错误

1. **一上来就全量看完所有 FN**
   - 不建议；先看第一批代表帧。

2. **只写口头感受，不写结构化字段**
   - 这样后面很难统计。

3. **看到框可疑就直接改原始标签**
   - 不建议；先做临时副本。

4. **所有难例都打成 `mixed_hard_case`**
   - 会失去分析价值。

5. **把“边界小人”误当成当前唯一主因**
   - 当前自动分桶已经明确说明，它不是主体。
