# person hardest FN 人工复核操作手册

## 1. 文档目的

本文档用于指导 `person_fullframe_with_new_labels` 这条线的 hardest FN 人工复核工作，重点回答下面几个问题：

1. 需要用什么软件；
2. 需要什么环境；
3. 软件怎么安装；
4. 实际复核时怎么一步一步做；
5. 什么时候只做“语义判断”，什么时候需要真正去修框或重标。

本文默认服务于当前目录：

- `backend-train-model/person-train-model/train-result/review/person_fullframe_with_new_labels_hard_sample_review/`

并与下面两个文件配合使用：

- `backend-train-model/person-train-model/train-docs/person分桶.md`
- `backend-train-model/person-train-model/train-result/review/person_fullframe_with_new_labels_prescreen_summary.json`

## 2. 当前人工复核要解决什么问题

当前自动分桶已经完成，下一步人工复核的目标不是重新跑模型，而是把 hardest FN 继续拆成更贴近业务语义的原因，例如：

- 暗光 / 背光
- 遮挡
- 半身 / 截断
- 姿态异常
- 远景内部小人
- 边界远景小人
- 拥挤 / overlap
- 可疑标注问题

因此人工复核的核心产出是：

1. 每个代表帧 / 每个 FN GT 的人工语义标签；
2. 是否怀疑原始标注需要修框或重标；
3. 哪一类问题最值得进入下一轮补样或增强策略。

## 3. 需要用到什么软件

## 3.1 必备软件

### 1. Visual Studio Code 或任意文本编辑器

用途：

- 编辑 `notes.md`
- 编辑 `semantic_bucket_summary.md`
- 查看 / 编辑 `semantic_bucket_manifest.json`

推荐：

- Visual Studio Code
- Notepad++

如果你平时就在 Cursor / VSCode 里工作，直接继续用当前编辑器就可以。

### 2. 图片查看软件

用途：

- 快速查看原图；
- 与 overlay 图对照；
- 不改框时，很多场景只靠图片查看器就够了。

可用：

- Windows 自带“照片”
- Honeyview
- FastStone Image Viewer

当前这一步不要求一定换专门软件，能稳定放大、切图、前后翻页即可。

### 3. LabelImg（当前最推荐）

用途：

- 查看 YOLO 检测框；
- 对可疑 bbox 做人工修框；
- 补充漏标 person；
- 删除明显错框。

为什么优先推荐 LabelImg：

- 当前任务本质上还是 **YOLO 检测框复核**；
- LabelImg 对矩形框和 YOLO txt 更直接；
- 对 person 单类检测场景足够简单。

## 3.2 可选软件

### 4. Labelme（可选，不是当前主推荐）

用途：

- 如果后续需要画更自由的可视化标注、区域说明、多边形辅助说明时可用；
- 但对当前“单类 person bbox 复核”来说，不如 LabelImg 直接。

当前建议：

- **默认优先用 LabelImg** 做 bbox 复核；
- **只有当你确实需要更自由的人工说明图** 时，再用 Labelme。

## 7. 人工复核分成两种模式

## 7.1 模式 A：只做语义复核，不改 bbox

适用场景：

- 你当前只想搞清楚 hardest FN 的语义原因；
- 暂时还不准备修改原始标签；
- 先做“这类问题到底是什么”的第一轮判断。

这时需要的软件最低配置是：

- 图片查看器
- 文本编辑器

输出主要写到：

- `sequence_*/notes.md`
- `semantic_bucket_manifest.json`
- `semantic_bucket_summary.md`

## 7.2 模式 B：语义复核 + 框级修正

适用场景：

- 你发现某个 GT 框明显有问题；
- 你怀疑存在漏标；
- 你需要真实修 YOLO 框，作为后续重训候选修订样本。

这时建议使用：

- LabelImg

并且先在当前 review 目录下做**临时工作副本**，不要上来就直接改原始标注。

## 8. 具体怎么做：第一轮人工复核详细步骤

## 8.1 第一步：先确定今天复核哪一批

不建议一上来全量看完所有 FN。

建议做法：

1. 先看 `person分桶.md`
2. 再看 `person_fullframe_with_new_labels_prescreen_summary.json`
3. 只选第一批 8 张代表帧开始

这样做的原因是：

- 成本低；
- 先抓最值钱样本；
- 很快就能判断当前主问题到底是什么。

## 8.2 第二步：进入对应序列目录

例如你先看 `D15_20260119061405`，就进入：

- `.../sequence_D15_20260119061405/`

如果你是按来源分工，而不是按序列分工，则建议先进入：

- `.../by_source/source_*/sequence_*/`

先看：

- `images/` 中的原图
- `labels/` 中的对应 txt
- `overlays/` 中不同 run 的 overlay 图

先看：

- `notes.md`

确认：

- 这条序列为什么被优先看；
- 当前建议优先看哪些帧；
- 当前待回答的问题是什么。

## 8.3 第三步：准备要看的图片

当前这一步**不需要**再把同一批素材复制到顶层 `sequence_*` 目录。

推荐做法是：

1. 直接使用 `by_source/source_*/sequence_*/images/` 中已经导出的原图；
2. 直接使用 `by_source/source_*/sequence_*/labels/` 中对应的 YOLO txt；
3. 直接使用 `by_source/source_*/sequence_*/overlays/` 中已经生成好的 overlay 图；
4. 顶层 `sequence_*` 目录只用来写 `notes.md`，不要再重复存第二份图片和 overlay。

这样做的好处是：

- 素材只维护一份；
- 组员不会在两个目录之间来回混淆；
- 后续如果要扩展导出范围，只需要继续维护 `by_source/`。

## 8.4 第四步：先做“纯语义判断”

打开原图和 overlay 图后，优先回答这些问题：

1. 这个人到底有多大？
2. 是不是明显暗光 / 背光？
3. 是不是被遮挡？
4. 是不是只露出半身？
5. 姿态是不是异常？
6. 是不是位于边界？
7. 是不是多人互相干扰？
8. GT 框本身是否可疑？

这一步先不要急着改框，先判断：

> 这个 FN 最主要是“语义难”还是“标注可疑”。

## 8.5 第五步：把结论记到 `notes.md`

建议先把人眼观察写到对应序列的 `notes.md`。

比如对某一帧，可以写成类似：

- `远景内部小人，人体对比度低，偏背光`
- `半身 + 遮挡，下半身被设备挡住`
- `边界小人，但本质还是像素不足`
- `GT 框偏松，怀疑需要修框`

这一层建议写中文短句，不要只写抽象标签。

## 8.6 第六步：把结构化字段写入 `semantic_bucket_manifest.json`

对于每个 FN GT，建议至少记录：

- `image_stem`
- `sequence_name`
- `split`
- `gt_index`
- `algorithm_bucket`
- `semantic_primary`
- `semantic_secondary`
- `need_relabel`
- `relabel_status`
- `notes`
- `reviewed_by`
- `reviewed_at`

推荐原则：

- `semantic_primary` 只写一个主因；
- `semantic_secondary` 只在确实需要时再写；
- `notes` 用中文写清楚肉眼依据。

## 8.7 第七步：只有当你确认框有问题，再进入 LabelImg 修框

这一步不是每张都必须做。

只有当你发现：

- 原始框明显偏移；
- 有 person 漏标；
- 某个框标成了不合理的范围；

才需要进入 LabelImg。

## 9. 使用 LabelImg 进行 bbox 复核的详细步骤

## 9.1 先准备一个临时工作目录

**不建议直接修改原始标签目录。**

推荐做法：

1. 在当前某个 `by_source/source_*/sequence_*/` 下，临时新建一个工作子目录，例如：
   - `relabel_work/`
2. 把要复核的：
   - `jpg`
   - 同名 `txt`
   - `classes.txt`
   复制进去。

其中 `classes.txt` 内容当前只需要一行：

```txt
person
```

这样做的好处是：

- 不会误伤原始标签；
- 修框前后可以对照；
- 后续确认无误后再决定是否回写正式标签。

## 9.2 启动 LabelImg

在终端里执行：

```powershell
conda activate yolo_code
labelImg
```

如果失败，再试：

```powershell
python -m labelImg
```

## 9.3 在 LabelImg 里设置 YOLO 模式

打开 LabelImg 后，注意下面几件事：

1. 将保存格式切换到 **YOLO**
2. `Open Dir` 选择你的临时工作目录
3. `Change Save Dir` 也指向这个临时工作目录
4. 确保类别只有 `person`

如果看到类别不对，不要继续乱存，先检查 `classes.txt`。

## 9.4 如何具体检查一张图

打开某张图片后，重点看：

1. 漏掉的 person 是否真的应该标；
2. 现有 bbox 是否明显过松 / 过紧；
3. 是否把背景、设备、反光区域一起框进去了；
4. 是否少标了半身或远景人；
5. 是否把其实不可判定的目标硬标成 person。

## 9.5 常用操作方式

以下是最常用的操作：

- 新建框：`W`
- 删除框：`Delete`
- 上一张：`A`
- 下一张：`D`
- 保存：`Ctrl + S`

调整框时：

- 用鼠标选中已有框；
- 拖动边或角点进行微调；
- 调整后立即保存。

## 9.6 修框后怎么记录

如果你在 LabelImg 里改了框，不要只改文件不记结论。

还需要同步记录：

- `need_relabel = true`
- `relabel_status = fixed` 或 `pending`
- `notes` 里写清楚改了什么，例如：
  - `原框过松，已收紧到人体边界`
  - `补标了右上角远景 person`
  - `原框误把设备一起框入`

## 10. 人工语义标签到底怎么打

## 10.1 当前建议使用的语义标签

建议优先使用这 9 个：

1. `dark_or_backlit`
2. `occluded`
3. `partial_body`
4. `pose_or_shape_unusual`
5. `small_far_interior`
6. `small_far_boundary`
7. `crowded_or_overlap`
8. `annotation_or_label_suspect`
9. `mixed_hard_case`

## 10.2 各标签怎么理解

### `dark_or_backlit`

适用：

- 人体与背景对比度低；
- 背光导致轮廓不清；
- 不是特别小，但视觉上不容易稳定识别。

### `occluded`

适用：

- 被设备、立柱、车辆、其他人挡住一部分；
- 目标可见性明显下降。

### `partial_body`

适用：

- 只露出上半身；
- 下半身出画；
- 被画幅截断；
- 人体完整性明显不足。

### `pose_or_shape_unusual`

适用：

- 弯腰；
- 蹲姿；
- 转身；
- 肢体形态不像标准直立 person。

### `small_far_interior`

适用：

- 小；
- 远景；
- 位于画面内部；
- 不是贴边导致的问题。

### `small_far_boundary`

适用：

- 小；
- 远景；
- 明显贴边或贴角。

### `crowded_or_overlap`

适用：

- 多人近邻；
- 两个框相互干扰；
- 人和人挤在一起；
- 很像定位 / 匹配不过线问题。

### `annotation_or_label_suspect`

适用：

- GT 框范围明显不合理；
- 是否该标本身就可疑；
- 很像需要回到原始标注再确认。

### `mixed_hard_case`

适用：

- 同时符合 2~3 类；
- 很难只归因到一个主因；
- 例如：`远景内部小人 + 背光 + 半身`。

## 10.3 主标签和次标签怎么写

建议规则：

- `semantic_primary`：只写一个最主要原因；
- `semantic_secondary`：如果确实存在第二个明显原因，再写；
- 如果你一上来就觉得“什么都占一点”，不要立刻全写 `mixed_hard_case`，先尽量找主因。

例如：

- 主因：`small_far_interior`
- 次因：`dark_or_backlit`

或者：

- 主因：`occluded`
- 次因：`partial_body`

## 11. 什么情况下要判定 `need_relabel = true`

建议只有下面几种情况才标 `true`：

1. 原始框明显偏移；
2. 框明显过大或过小；
3. 明显漏标 person；
4. GT 本身是否该标都可疑；
5. 同一类样本在多个帧里都出现类似标注异常。

## 11.1 `relabel_status` 怎么写

- `not_needed`
  - 不需要修标。
- `suspected`
  - 怀疑有问题，但还没真正改。
- `fixed`
  - 已在临时工作目录修完。
- `pending`
  - 确认要改，但还没动手。

## 12. 第一轮人工复核完成后，需要更新哪些文件

第一轮看完后，相关记录主要落在这三个地方：

1. `sequence_*/notes.md`
   - 写序列级观察结论。
2. `semantic_bucket_manifest.json`
   - 写逐 GT 结构化记录。
3. `semantic_bucket_summary.md`
   - 写全局可读总结。

如果是**单人**复核，建议顺序：

> 先写 `notes.md` 和 `manifest.json`，最后再写 `summary.md`。

如果是**多人分工**，推荐顺序改成：

> 组员先写各自负责序列的 `notes.md` ->
> 再由汇总人批量合并 `semantic_bucket_manifest.json` ->
> 最后由组长或统一汇总人更新 `semantic_bucket_summary.md`。

这样做的原因是：

- `notes.md` 是按序列拆开的，适合多人并行；
- `semantic_bucket_manifest.json` 是单个 JSON 文件，多人同时改最容易冲突；
- `semantic_bucket_summary.md` 是汇总结论，不适合每看完一张图就实时改。

另外：

- `by_source/review_asset_manifest.json` 是这轮素材导出清单；
- 它只负责说明“导出了哪些图、标签和 overlay”；
- **人工复核过程中不需要改这个文件。**

## 13. 组员统一填写规范

## 13.1 组员各自应该改哪些文件

推荐按下面口径执行：

### 13.1.1 可以直接发给组员的最简规则

如果你要把任务直接分给组员，最简单、最不容易出错的规则就是：

#### 组员默认只需要做两件事

1. 在 `by_source/` 里看自己负责的素材
2. 在对应的 `sequence_*/notes.md` 里写观察结论

也就是说，**组员默认主要修改的是各自负责序列的 `notes.md`**。

#### 组员默认不要改的文件

除非你额外指定了专人负责汇总，否则组员**不要直接改**下面这些文件：

- `semantic_bucket_manifest.json`
- `semantic_bucket_summary.md`
- `by_source/review_asset_manifest.json`

原因是：

- `semantic_bucket_manifest.json` 是全局共用的单个 JSON，很多人同时改最容易冲突；
- `semantic_bucket_summary.md` 是最终总结文件，不适合每个组员边看边改；
- `by_source/review_asset_manifest.json` 只是素材导出索引，不是人工结论记录表。

#### 谁负责改哪些文件

建议明确分工如下：

- **普通组员**：只改自己负责序列的 `notes.md`
- **汇总人**：统一整理并更新 `semantic_bucket_manifest.json`
- **组长 / 最终汇总人**：统一更新 `semantic_bucket_summary.md`
- **素材维护人**：只有当重新导出或扩充素材时，才处理 `by_source/review_asset_manifest.json`

如果你不想让组员自己碰 JSON，那么可以要求他们：

> 只提交 `notes.md` 内容或按照统一表格把结果发给汇总人，
> 然后由汇总人统一更新 `semantic_bucket_manifest.json`。

### 组员日常复核时

优先修改：

1. 自己负责序列的 `sequence_*/notes.md`
2. 交给汇总人的结构化记录内容

不建议每个组员每看完一张图就直接去改：

- `semantic_bucket_summary.md`

因为这个文件是全局总结，应该按批次统一更新。

### 汇总阶段

由指定汇总人统一更新：

1. `semantic_bucket_manifest.json`
2. `semantic_bucket_summary.md`

### 明确不需要改的文件

- `by_source/review_asset_manifest.json`

这个文件只是素材导出索引，不是人工结论记录表。

### 13.1.2 推荐给组员的明确口径

为了避免误解，可以直接把下面这段话发给组员：

> 你只需要在 `by_source/` 里看分给你的图片、标签和 overlay，
> 然后把观察结果写到对应序列的 `notes.md`。
> `semantic_bucket_manifest.json`、`semantic_bucket_summary.md`、`by_source/review_asset_manifest.json` 默认不要自己改，
> 由指定汇总人统一处理。

## 13.2 每条结构化记录怎么填

往 `semantic_bucket_manifest.json` 写一条记录时，建议按下面规则：

### `image_stem`

- 直接写图片 stem，不带扩展名；
- 必须与素材文件名一致。

示例：

- `D15_20260119061405_frame_0345`

### `sequence_name`

- 写该图片所属序列；
- 必须与 `notes.md` 所在序列目录一致。

示例：

- `D15_20260119061405`

### `split`

- 只允许写：`val` 或 `test`
- 可从 overlay 文件名对应的 run 推断，或从现有复盘结果中确认。

### `gt_index`

- 写该图片里这个 FN GT 的索引；
- 如果当前只先做帧级判断、还没精确到 GT 索引，不要乱填，先留待汇总人补。

### `algorithm_bucket`

- 只写当前 4 主桶之一：
  - `small_boundary_person`
  - `small_interior_person`
  - `medium_large_pose_or_appearance`
  - `crowded_or_localization`

### `semantic_primary`

- 只写一个主语义标签；
- 当前优先从下面这组里选：
  - `dark_or_backlit`
  - `occluded`
  - `partial_body`
  - `pose_or_shape_unusual`
  - `small_far_interior`
  - `small_far_boundary`
  - `crowded_or_overlap`
  - `annotation_or_label_suspect`
  - `mixed_hard_case`

### `semantic_secondary`

- 可空；
- 只有确实存在第二个明显原因时再写；
- 没有就写空字符串 `""`。

### `need_relabel`

- 只写布尔值：`true` / `false`
- 含义：这条样本是否怀疑需要修框、补标或回查原标注。

### `relabel_status`

- 只允许写：
  - `not_needed`
  - `suspected`
  - `fixed`
  - `pending`

推荐规则：

- 不需要改 -> `not_needed`
- 怀疑有问题但还没动 -> `suspected`
- 已在临时副本修完 -> `fixed`
- 确认该改但还没改 -> `pending`

### `notes`

- 用中文短句写肉眼依据；
- 不要只写“难”“小”“看不清”。

推荐写法：

- `远景内部小人，人体对比度低，偏背光`
- `半身 + 遮挡，下半身被设备挡住`
- `边界小人，但本质仍像素不足`
- `GT 框偏松，怀疑需要修框`

### `reviewed_by`

- 写复核人姓名、缩写或工号；
- 组内保持统一格式即可。

### `reviewed_at`

- 统一写 `YYYY-MM-DD`

示例：

- `2026-05-06`

## 13.3 `notes.md` 建议怎么写

`notes.md` 不是结构化表，它更适合写：

1. 这条序列主问题是什么；
2. 哪几张帧最有代表性；
3. 为什么判断成某类语义问题；
4. 是否怀疑存在标注问题；
5. 下一步建议动作。

建议每次至少写到这 3 类信息：

- **观察对象**：看了哪些帧
- **判断结论**：更偏哪类语义问题
- **是否需要修标**：有 / 无 / 待确认

## 13.4 推荐的团队协作节奏

如果你把任务分给多位组员，建议这样分工：

1. **组员 A/B/C**
   - 直接在 `by_source/` 里看图；
   - 各自负责某几个来源或序列；
   - 把结论先写到自己负责的 `sequence_*/notes.md`。

2. **汇总人**
   - 定期把组员确认过的结论合并到 `semantic_bucket_manifest.json`；
   - 检查字段拼写是否统一。

3. **组长 / 最终汇总人**
   - 在一批样本看完后，再更新 `semantic_bucket_summary.md`；
   - 输出阶段性结论和下一步建议。

这样最稳，因为：

- 多人同时改一个 JSON 容易冲突；
- 多人同时改 summary 也容易相互覆盖；
- `notes.md` 按序列拆开，更适合并行。

## 14. 当前最容易犯的错误

1. **一上来就全量看完所有 FN**
   - 不建议；先看第一批代表帧。

2. **只写口头感受，不写结构化字段**
   - 这样后面很难统计。

3. **看到框可疑就直接改原始标签**
   - 不建议；先做临时副本。

4. **所有难例都打成 `mixed_hard_case`**
   - 会失去分析价值。

5. **把“边界小人”误当成当前唯一主因**
   - 当前自动分桶已经明确说明，它不是主体。
