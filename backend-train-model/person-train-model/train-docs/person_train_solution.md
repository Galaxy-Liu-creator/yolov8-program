## 2026-05-05 基于正式 `train768 + eval768` 结果的结论修正与后续顺序调整



### 1. 本轮修订目的



本轮记录用于修正上一条 `2026-05-04` 记录中三类已经被新证据覆盖的表述：

- `person_fullframe_with_new_labels_img768` 的外部 eval 数值；
- 对 `img768` 评估口径问题的表述强度；
- 下一轮推荐执行顺序。

本轮依据的主要证据为：

- `train-result/artifacts/runs/person_fullframe_with_new_labels_baseline/results.csv`
- `train-result/artifacts/runs/person_fullframe_with_new_labels_img768/results.csv`
- `train-result/artifacts/reports/person_fullframe_with_new_labels_baseline_train.json`
- `train-result/artifacts/reports/person_fullframe_with_new_labels_img768_train.json`
- `train-result/artifacts/reports/person_fullframe_with_new_labels_baseline_eval.json`
- `train-result/artifacts/reports/person_fullframe_with_new_labels_img768_eval.json`



### 2. 当前观察到的问题



当前围绕 `person_fullframe_with_new_labels` 这条 fullframe 扩样线，最新已经可以确认三件事：

1. `imgsz=768` 并不是完全没有收益；它对高 IoU 指标有**小幅真实提升**。
2. 但这份提升仍然没有把当前 `fullframe + yolov8n + 扩样数据` 组合的高 IoU ceiling 明显抬高。
3. 当前主矛盾仍然是：
   - `mAP50` 不低；
   - `precision / recall` 也不算差；
   - 但 `mAP75 / mAP50-95` 明显更难继续提升；
   - 说明瓶颈更像是**高 IoU 下的框定位精度**，而不是“完全检测不到人”。



### 3. 关键证据与直接判断



#### 3.1 当前正式对齐口径的外部 eval 已经完成



当前应以这两份外部 eval 报告为准：

- `person_fullframe_with_new_labels_baseline_eval.json`
  - val `mAP50-95 = 0.50948`
  - test `mAP50-95 = 0.48018`

- `person_fullframe_with_new_labels_img768_eval.json`
  - val `mAP50-95 = 0.51466`
  - test `mAP50-95 = 0.49699`

这意味着：

- 上一条记录里写的 `img768` 外部 eval
  - val `0.49433`
  - test `0.48893`

已经不是当前最新、最可信的正式结论，后续分析不应再沿用这组旧数值。



#### 3.2 `img768` 的真实收益存在，但主要落在更高 IoU 区间



把 baseline 与当前正式 `img768` eval 对齐后，可以看到：

- val：
  - `mAP50: 0.95170 -> 0.94568`，略降
  - `mAP75: 0.49104 -> 0.51355`，提升
  - `mAP50-95: 0.50948 -> 0.51466`，小幅提升

- test：
  - `mAP50: 0.90537 -> 0.89480`，略降
  - `mAP75: 0.45269 -> 0.49676`，提升
  - `mAP50-95: 0.48018 -> 0.49699`，小幅提升

这说明 `768` 的作用更像是：

> 帮一部分样本把框贴得更准了，所以 `mAP75` 提升比较明显；但这种收益还不足以在更高 IoU 全区间内形成大幅跃升，因此 `mAP50-95` 只表现为小幅改善，而不是跨台阶突破。



#### 3.3 `img768` 可以明确判定为平台期早停；baseline 只能判定为后段已平台化



- `img768` 的 `results.csv` 显示：
  - 最佳 train-time val `mAP50-95` 出现在 **epoch 29**，约为 `0.51352`；
  - 最后一行停在 **epoch 69**；
  - `69 - 29 = 40`，与 `patience=40` 完全对齐。

因此，`img768` 这次停止可以明确视为：

> 正常的 patience 平台期早停，而不是训练还在持续明显上升却被过早截断。

- baseline 的 `results.csv` 显示：
  - train-time val `mAP50-95` 较高点出现在 **epoch 52**，约为 `0.51030`；
  - 当前保留产物最后停在 **epoch 155**。

- 但 `person_fullframe_with_new_labels_baseline_train.json` 同时显示：
  - 这次 baseline 训练是 **resume run**；
  - 初始化来源是本 run 自己的 `weights/last.pt`。

因此，对 baseline 更严谨的表述应改为：

> 当前证据支持 baseline 后段已经平台化，没有显示出“只要再训就会明显继续涨”的迹象；但由于这是 resume run，不能像 `img768` 一样，把它精确还原为一次单独、干净的标准 patience 早停事件。



#### 3.4 当前这次 `img768` 正式 eval 已经不是“按 640 误评估”



`person_fullframe_with_new_labels_img768_eval.json` 的 `runtime.argv` 已明确记录：

- `--imgsz 768`
- `--batch 4`
- `--workers 4`

因此，当前这份正式 `img768` 指标：

> 不是因为评估阶段误回退到 `640` 才导致结论失真。

更准确的说法应是：

- 代码机制上，如果评估命令漏传 `--imgsz 768`，确实可能回退到默认 `640`；
- 但当前已经保留的正式 `img768` eval 报告，属于 **train `768` + eval `768`** 的对齐口径；
- 所以现在继续把“640 误评估”当成当前主解释，已经不合适。

需要额外提醒的是：`person_fullframe_with_new_labels_img768_train.json` 里记录的 `project_config_path` 仍是 `person_project_config.json`，而 `dataset_yaml` 又明确指向 `person_fullframe_with_new_labels` prepared 数据集。这说明 `img768` 的 train report 元数据存在**配置口径不自洽**的问题，但这更像是**报告元数据噪声**，不是当前 `mAP50-95` ceiling 的主因。



### 4. 原因排序



#### 4.1 第一主因：高 IoU 框定位精度不足，而不是“完全检测不到人”



这是当前证据最强的判断。

因为两个 run 都表现为：

- `mAP50` 不低；
- `precision / recall` 不算差；
- 但 `mAP75 / mAP50-95` 提升明显更难。

这说明当前更像是：

> 模型多数时候能知道“这里有人”，但在暗光、小目标、远景、遮挡样本上，框没有稳定贴到足够高的 IoU 区间。



#### 4.2 第二主因：fullframe 路线天然不利于远景小目标的高精度回归



当前这条线仍是 fullframe person：

- 小目标缩放后有效像素有限；
- 框只要偏几像素，高 IoU 指标就会掉得很快；
- `768` 虽然能补一点像素，但还不足以从根本上改变这个问题。

因此当前的本质瓶颈不是“有没有检出来”，而是：

> **对扩样后的远景小人、遮挡人、半身人和复杂背景人，bbox 贴合度不够稳定。**



#### 4.3 第三主因：扩样后难样本占比上升，使验证集更早碰到高 IoU ceiling



当前更准确的结论不是：

> 因为图片从 `500+` 变成 `3000+`，所以自然更容易早停。

而是：

> 扩样后加入了更多暗光、小目标、远景、遮挡、半身、复杂背景样本，使模型更快暴露出高 IoU 定位上限。

这也是为什么：

- baseline 的 train-time best 约 `0.51030`
- `img768` 的 train-time best 约 `0.51352`

两者都大致卡在 `0.51` 这一带。



#### 4.4 第四主因：`imgsz=768` 带来的是有限增益，不是 ceiling 级突破



当前正式结果已经说明：

- `768` 不是没用；
- 但它带来的收益主要是**部分样本的高 IoU 框质量改善**；
- 它没有把当前组合的上限明显抬高一截。

因此它更像：

> **更快摸到上限，并在高 IoU 区间小幅改善，而不是彻底突破当前 ceiling。**



#### 4.5 第五主因：`yolov8n` 容量可能开始吃紧，但目前仍属于次主因



当前数据已经扩展到：

- `3009` 张图片
- `8861` 个框

而且新增样本难度更高。在这种情况下，`yolov8n` 可能出现：

- recall 还能维持；
- `mAP50` 也不算差；
- 但 `mAP75 / mAP50-95` ceiling 开始偏低。

不过当前仅凭现有证据，还不适合把“模型容量不够”排到第一主因。它更像是：

> 在数据难度、fullframe 小目标限制之后，再考虑的下一层变量。



#### 4.6 第六主因：报告元数据不一致会干扰解读，但不是当前指标上不去的主解释



`img768` 的 train report 元数据中，`project_config_path` 与实际 prepared dataset 口径不完全自洽，这确实会干扰历史复盘。

但当前正式 `img768` eval 报告已经明确按 `768` 完成，因此：

> 元数据问题是分析噪声，不是当前 `mAP50-95` ceiling 的主原因。



### 5. 对上一条记录的修正



相对 `2026-05-04` 的上一条记录，本轮需要明确修正三点：

1. `img768` 的当前正式外部 eval 不再是：
   - val `0.49433`
   - test `0.48893`

   而应改为：
   - val `0.51466`
   - test `0.49699`

2. “上一轮 `img768` 评估命令漏传 `--imgsz 768`，因此评估按 `640` 跑”这句话，当前只能保留为：
   - **历史上可能发生过的命令口径风险**；
   - 不能继续当成当前正式结论的主解释。

3. 关于早停的表述应区分强弱：
   - `img768`：可以明确写成平台期 patience 早停；
   - baseline：应写成“后段已平台化，但由于是 resume run，不宜直接写成和 `img768` 同等强度的标准早停证据”。



### 6. 修正后的推荐执行顺序



当前更稳妥的下一轮顺序建议改成：

1. **先锁定当前正式对照结论，不再把重点放在重复补同一版 `img768` 评估。**
   - 当前已经有 `train 768 + eval 768` 的正式报告；
   - 后续复盘应基于这份对齐结果，而不是继续把“是否误按 640 评估”当作第一优先级。

2. **对 hardest val/test 样本做分桶复盘。**
   - 重点按：暗光 / 背光、小目标 / 远景、遮挡、半身、复杂背景、靠边框偏移 分桶；
   - 先搞清楚：到底是哪一类样本主导了高 IoU 指标上不去。

3. **如果小目标 / 远景占主导，优先尝试局部裁切、切片，或后续补齐 ROI 后再做 ROI-aware 扩样版本。**
   - 当前不建议把主要希望继续押在单纯放大 `imgsz` 上；
   - 因为 `768` 已经证明：它只有有限增益，没有形成明显突破。

4. **如果暗光 / 低对比度占主导，优先做暗光专项补强。**
   - 包括定向补样；
   - 亮度 / 对比度增强微调；
   - 必要时做轻量低照度预处理对照。

5. **在前面原因已经分清后，再做单因子的更高容量模型对照。**
   - 例如 `yolov8s`；
   - 但前提是先避免把数据难度、fullframe 路线、分辨率和模型容量混在一轮里一起改。

6. **如果后续需要更严谨地审查停训机制或稳定性，再补 seed 对照或补一版干净的非 resume baseline。**
   - 这一项可以做；
   - 但它不是当前第一优先级。



### 7. 当前不建议优先做的事



#### 7.1 不建议继续把主要希望放在“多训一点”或“再加 patience”上



当前证据更像是：

- 已经摸到验证集高 IoU ceiling；
- 而不是“纯粹没训够”。



#### 7.2 不建议只看 `mAP50`



这条线最容易误判的点就是：

- `mAP50` 看起来不错；
- 但真正卡住的是 `mAP75 / mAP50-95`。



#### 7.3 不建议把“历史上可能有 640 误评估”继续当作当前主线解释



当前已经保留的正式 `img768` eval 报告就是按 `768` 跑的，因此现在再把主要精力放在“是不是评估口径错了”，优先级已经过高。



#### 7.4 不建议同一轮同时改太多变量



不要在同一轮里同时改：

- `imgsz`
- `batch`
- 模型大小
- 增强策略
- 数据构成

否则后面很难知道到底是哪一个变量带来了收益。



### 8. 本轮结论边界



- 当前可以明确判断：`img768` 已经在正式 `train 768 + eval 768` 口径下，表现出**小幅真实的高 IoU 指标收益**，但没有形成 ceiling 级突破。
- 当前可以明确判断：当前正式 `img768` eval 报告不是按 `640` 误评估，因此“评估口径错误”不能再作为当前主解释。
- 当前也可以明确判断：两次 run 的共同主瓶颈更像是高 IoU 定位精度，而不是 recall 崩掉。
- 当前还不能只凭现有结果，就把“模型容量不足”直接升级成第一主因；在这之前，仍应先完成 hard sample 分桶复盘。



### 9. 当前一句话结论



当前 `person_fullframe_with_new_labels` 这条线更像是：

> `imgsz=768` 已经证明自己能在高 IoU 区间带来一点真实收益，但 fullframe + `yolov8n` 在扩样后的暗光、小目标、远景和复杂场景上，主要瓶颈仍然是框定位精度 ceiling，而不是当前正式评估口径错误或单纯没训够。



## 2026-05-04 `person_fullframe_with_new_labels_baseline` / `img768` 早停原因分析与解决方案



### 1. 当前观察到的问题



当前围绕 `person_fullframe_with_new_labels` 这条 fullframe 扩样线，出现了两个值得重点解释的现象：



1. `person_fullframe_with_new_labels_baseline` 没有跑满 `180` 轮；

2. `person_fullframe_with_new_labels_img768` 也较早停止，并且更早进入平台期；

3. 两次训练都表现出：

   - `mAP50` 不低；

   - 但 `mAP75 / mAP50-95` 提升有限；

   - 说明瓶颈更像是**高 IoU 下的框定位精度**，而不是“完全检测不到人”。



当前两个关键外部 eval 结果为：



- `person_fullframe_with_new_labels_baseline`

  - val `mAP50-95 = 0.50948`

  - test `mAP50-95 = 0.48018`

- `person_fullframe_with_new_labels_img768`

  - val `mAP50-95 = 0.49433`

  - test `mAP50-95 = 0.48893`



### 2. 关键证据与直接判断



#### 2.1 这两次“早停”更像平台期，而不是单纯训练不够



- `person_fullframe_with_new_labels_img768` 的 train-time val `mAP50-95` 最佳点出现在 **epoch 29**，约为 `0.51352`；

- 此后到 **epoch 69**，都没有再刷新 best，因此耗尽了 `patience=40`。



- `person_fullframe_with_new_labels_baseline` 的全局较高点出现在更靠后阶段，train-time val `mAP50-95` 大约在 **epoch 52** 达到 `0.51030`；

- 后续即使 resume 后继续训练，也没有稳定突破这一量级，说明它并不是“还能明显继续涨，只是提前停了”，而是**已经在验证集上碰到了相近的 ceiling**。



#### 2.2 两次训练都卡在差不多的验证集 ceiling



- `img768` 的 train-time best 约 `0.51352`

- baseline 的 train-time best 约 `0.51030`



这两个数非常接近，说明问题不是偶然，而是：



> 当前这份扩样后的 fullframe 数据，在 `yolov8n + fullframe + 当前训练策略` 组合下，验证集高 IoU 指标本身就大约卡在 `0.51` 左右。



### 3. 原因排序



#### 3.1 第一主因：新增数据显著提高了任务难度，而不是单纯“图片变多”



当前更准确的结论不是：



> 因为从 `500+` 图变成 `3000+` 图，所以更容易早停。



而是：



> 扩样后加入了更多**暗光、小目标、远景、遮挡、半身、复杂背景**样本，使得模型在验证集上的高 IoU 定位能力更早碰到上限。



也就是说，真正起作用的是**难样本占比提升**，而不是图片数量本身。



#### 3.2 第二主因：fullframe 路线天然不利于远景小目标的高精度框回归



当前这条线仍是 fullframe person：



- 对小目标来说，缩放后有效像素有限；

- 模型往往能“看到人”，但不一定能“框得很准”；

- 这会直接表现为：

  - `mAP50` 还行；

  - `mAP75 / mAP50-95` 很快进入平台。



因此，两次早停共同暴露出的本质问题，不是“有没有检出来”，而是：



> **对扩样后的暗光、小目标、远景、遮挡样本，bbox 贴合度不够。**



#### 3.3 第三主因：`imgsz=768` 只是更快吃到分辨率红利，但没有突破根本 ceiling



`img768` 这条线的特点是：



- 前 20~30 轮提升更快；

- 更早到达 train-time val best；

- 后面仍然无法继续显著刷新。



这说明 `768` 的确帮助模型更早获得小目标细节收益，但并没有从根本上解决：



- 暗光边界不清；

- 遮挡导致的框偏移；

- fullframe 下远景人过小；

- 小框 IoU 对像素偏差极其敏感。



因此它更像：



> **更快摸到上限**，而不是**把上限抬高很多**。



#### 3.4 第四主因：上一轮 `img768` 评估命令漏传了 `--imgsz 768`，导致评估实际按 640 运行



这是一个**命令口径问题**，不是训练本身的问题。



根因已经从 `run_person_flow.py` 查到：



- `parse_args()` 中 `--imgsz` 的默认值是 `640`；

- `evaluate_command()` 会直接把 `args.imgsz` 传给底层评估命令；

- 如果评估 CLI 没有显式写 `--imgsz 768`，那就会自动回退到默认值 `640`。



因此，上一轮 `img768` 的评估报告里显示 `640`，不是训练报告写错，而是：



> 之前给出的 `person_run_method.md` 评估命令漏传了 `--imgsz 768`，导致评估阶段实际使用了默认 `640`。



这会带来一个额外后果：



- 训练时用 `768` 学到的一部分小目标细节；

- 在 `640` 推理评估时被重新压缩；

- 导致 `img768` 的真实收益没有被完整兑现。



#### 3.5 第五主因：`yolov8n` 容量在更难的数据集上已经开始吃紧



当前数据已经从旧版 `502` 图扩展到：



- `3009` 张图片

- `8861` 个框



而且新增的并不是大量简单样本，而是更复杂、更难定位的样本。此时 `yolov8n` 很容易出现：



- recall 还能维持；

- `mAP50` 也不算差；

- 但 `mAP50-95` 与 `mAP75` 上限开始吃紧。



所以模型容量更像是**次主因**，不是最先要动的变量，但已经进入候选问题列表。



### 4. 对应解决方案



#### 4.1 第一优先级：先修正评估口径，再重新做 `img768` 对照



当前最直接必须修正的一件事是：



- `img768` 的评估必须显式传 `--imgsz 768`



原因：



- 否则 `run_person_flow.py evaluate` 会回退到默认 `640`；

- 这样会把“训练分辨率提升”的收益部分抵消掉；

- 也会让后续分析混进新的口径噪声。



也就是说，后续应明确区分：



1. train `768` + eval `768`

2. train `768` + eval `640`

3. train `640` + eval `640`



先把训练/评估分辨率口径理清，再谈下一步结论。



#### 4.2 第二优先级：做 hard sample 分桶复盘，而不是泛泛继续扩样



当前最值得补的，不是普通 person，而是：



- 暗光 / 背光

- 小目标 / 远景

- 遮挡

- 半身

- 密集背景

- 靠边、容易框偏的样本



建议把 val/test 里表现差的样本按这些桶分开，再看哪一类占主导。只有分桶后，才能判断：



- 主要是小目标问题；

- 还是暗光问题；

- 还是遮挡 / 半身问题。



#### 4.3 第三优先级：如果小目标 / 远景占主导，优先考虑局部裁切或切片



如果复盘后确认：



- 大多数高 IoU 误差都集中在远景小人；

- 而且这些目标在 fullframe 下始终很小；



那优先方案不应只是继续堆全图训练，而应该考虑：



- 局部 crop 训练；

- 切片训练 / 切片推理；

- 或后续补齐 ROI 后，准备新的 ROI-aware 扩样版本。



#### 4.4 第四优先级：如果暗光 / 低对比度占主导，做暗光专项补强



如果复盘后发现主要问题集中在暗光场景，那么更合理的动作是：



- 定向补充暗光人样本；

- 微调亮度 / 对比度相关增强；

- 必要时做轻量低照度预处理对照。



这类问题靠“单纯继续训更久”往往解决不了。



#### 4.5 第五优先级：在前面结论清楚后，再考虑更高容量模型



如果下面几步都验证后，`mAP50-95` 仍明显卡住：



1. 分辨率评估口径已修正；

2. hard sample 已分桶；

3. 小目标 / 暗光等主因已确认；



再考虑更高容量模型，例如：



- `yolov8s`



当前不建议一上来先切更大模型，因为那会把：



- 数据难度问题

- fullframe 问题

- 分辨率问题

- 模型容量问题



一起混掉。



### 5. 当前不建议优先做的事



#### 5.1 不建议把主要希望放在继续增加 epochs 或单纯调大 patience



两次训练已经说明：



- train loss 还能继续降；

- 但 val `mAP50-95` 很难继续涨。



这说明当前瓶颈更像：



- 数据难度；

- 小目标像素；

- fullframe 定位上限；



而不是“纯粹没训够”。



#### 5.2 不建议只看 `mAP50`



当前这条线最容易误判的点，就是：



- `mAP50` 看起来不错；

- 但 `mAP75 / mAP50-95` 才真正反映小目标和复杂场景下的框质量。



#### 5.3 不建议一次同时改太多变量



不要在同一轮里同时改：



- `imgsz`

- `batch`

- 训练 / 推理分辨率

- 模型大小

- 增强策略

- 数据构成



否则后面很难知道到底是谁带来了收益。



### 6. 推荐执行顺序



当前更稳妥的下一轮顺序建议写成：



1. 先按修正后的命令，重新做 **train `768` + eval `768`**；

2. 再与 **train `640` + eval `640`** 对照；

3. 对 hardest val/test 样本做分桶复盘；

4. 如果小目标 / 远景占主导，优先试局部裁切 / 切片；

5. 如果暗光占主导，优先补暗光专项样本与增强；

6. 最后再考虑更高容量模型。



### 7. 本轮结论边界



- 当前可以明确判断：**两次早停的主因不是“图片数量变多本身”，而是扩样后难样本占比上升，使模型在高 IoU 验证指标上更早进入平台期。**

- 当前也可以明确判断：上一轮 `img768` 评估之所以显示 `640`，是因为 `person_run_method.md` 中的评估命令漏传了 `--imgsz 768`，而 `run_person_flow.py evaluate` 的 CLI 默认值正好是 `640`。

- 当前还不能直接把 `img768` 的全部真实潜力定死，因为上一轮评估口径本身就没有完全对齐训练分辨率。



### 8. 当前一句话结论



当前 `person_fullframe_with_new_labels` 这条线更像是：



> 扩样后新增的大量暗光、小目标、远景和复杂场景样本，把 fullframe + `yolov8n` 的高 IoU 定位上限提前暴露出来了；而上一轮 `img768` 的评估命令又漏传了 `--imgsz 768`，导致它的收益没有被完整、公平地评估出来。
