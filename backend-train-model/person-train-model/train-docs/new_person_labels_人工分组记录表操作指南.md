 本文档用于指导组员正确使用下面这份 Excel 模版：

1. Excel 模版中**每一个标题**分别是什么意思；

## 4. `填写说明` 工作表：每个标题和模块是什么意思

 这个工作表的两个列标题是：


| 标题   | 意义         |
| ---- | ---------- |
| `模块` | 当前这一行要讲的主题 |
| `说明` | 对该主题的详细解释  |


 这个工作表里当前已经有若干说明模块。下面逐条解释它们的意义：

### 4.1 `总体目标`

 意思是：

- 这份表的用途是做 `new_person_labels` 的人工逻辑分组；
- 同时确认每组应该采用什么 ROI 策略；
- 并把后续的分工协作记下来。

 组员要理解为：

> 这不是单纯“看图备注”，而是把自动组真正整理成后续可进入 ROI-aware 的人工最终组。

### 4.2 `当前主参考版本`

 意思是：

- 当前默认以 `t006` 自动粗分组结果作为主参考；
- 不是 `t008`，也不是 `t004`。

 组员要理解为：

> 没有特殊说明时，都先围绕 `t006` 的 `group_0001 ~ group_0006` 开始判断。

### 4.3 `填写顺序`

 意思是：

- 先看组级结果；
- 再补图片明细；
- 最后做 ROI 标注计划。

 组员不要反着来，不建议一上来就逐图填 2500 多张图。

### 4.4 `A类稳定组`

 意思是：

- 同摄像头；
- 同视角；
- 背景和作业区边界稳定；
- 大多数图可共用一版 ROI。

 组员只要判断某组符合这些条件，就应优先把它标成 `A`，并尽快推进组级 ROI。

### 4.5 `B类半稳定组`

 意思是：

- 大体同场景；
- 但仍有轻微裁切变化、光照差异或构图差异。

 组员遇到这种组，不要马上逐图 ROI，优先先写成 `B`，再继续细拆。

### 4.6 `C类不稳定组`

 意思是：

- 视角明显变化；
- ROI 边界明显变化；
- 或混有多个场景。

 组员遇到这类组，通常两种处理方式：

- 逐图 ROI；
- 或暂缓纳入 ROI-aware 第一版。

### 4.7 `ROI策略填写`

 意思是：后面表格里的 ROI 策略字段，统一只使用以下值：

- `组级ROI`
- `逐图ROI`
- `暂缓`

 组员不要自己发明新说法，例如：

- `按组标`
- `先不做`
- `可能逐图`

 这会导致统计口径混乱。

### 4.8 `自动粗分组说明`

 意思是：

- 当前 `t006` 一共有 `6` 个自动组；
- 其中 `group_0002 / group_0005 / group_0006` 应优先复核；
- 这是大家开始分工时的重点提示。

### 4.9 `最终产物`

 意思是：

- 最终确认后的稳定组，应落到：
  - `backend-train-model/person-train-model/train-result/working/new_person_labels_frames_grouping_t006/grouped/`
- 然后再进入代表帧抽取和 ROI 标注。

 组员要理解为：

> Excel 不是最终产物，Excel 只是把最终要落到工作目录里的组先管理清楚。

## 5. `组级台账` 工作表：每一个标题的意义

这是主表。每一行建议先对应一个**自动组**；如果后续确认这个自动组内部还混有多个稳定小场景，再在它下面扩展成多个**人工最终组**。

要特别注意：

> **继续细拆的依据，不是只看代表帧本身，而是“代表帧先发现问题 + 组内再抽样验证 + 这个差异确实会影响是否能共用 ROI”。**

也就是说，代表帧是第一轮入口，但不是唯一依据。

 下面按标题逐项解释。


| 标题                   | 意义                             | 怎么填                               | 示例                    |
| -------------------- | ------------------------------ | --------------------------------- | --------------------- |
| `当前自动组ID（来自t006）`    | 当前这一行对应哪个自动粗分组                 | 直接填 `group_0001` 这类 ID            | `group_0002`          |
| `当前组图片数`             | 当前自动组或人工最终组的图片数量               | 初始可沿用自动组数量；细拆后填子组实际数量或待统计         | `449`                 |
| `当前组分辨率`             | 当前组图片的统一分辨率                    | 用 `宽x高` 格式                        | `1920x1080`           |
| `人工细分后的最终组ID`        | 最终准备拿去做 ROI 判断和后续落盘的人工组 ID     | 建议短、稳定、可追踪                        | `group_0002_a_day`    |
| `人工组名称/场景名`          | 便于人理解的中文名字                     | 建议写“场景 + 视角 + 条件”                 | `白天近景左侧视角`            |
| `摄像头编号（未知可留空）`       | 如果能判断出摄像头来源，就记录                | 不能确定可先空着                          | `cam01`               |
| `场景编号/区域名`           | 场景或区域标识                        | 写区域名、作业区名即可                       | `zoneA`               |
| `视角编号`               | 主作业区 / 计划 ROI / 主要目标簇在画面中的横向位置 | 优先填 `左侧 / 居中 / 右侧` 这类统一口径，而不是随意编号 | `居中`                  |
| `稳定性等级：A/B/C`        | 当前组属于稳定、半稳定还是不稳定               | 只填 `A`、`B`、`C`                    | `B`                   |
| `组级ROI/逐图ROI/暂缓`     | 当前组采用哪种 ROI 策略                 | 只填这三种固定值                          | `组级ROI`               |
| `是否可共用一版ROI`         | 当前组能否共享一版 ROI polygon          | 建议统一填 `是 / 否 / 待定`                | `是`                   |
| `是否需要继续细拆`           | 当前组是否还要继续拆成子组                  | 建议统一填 `是 / 否 / 待定`                | `是`                   |
| `继续细拆依据（如白天/夜晚/远近景）` | 如果需要细拆，按什么维度拆                  | 写最主要的细拆依据                         | `白天/夜晚`               |
| `代表帧文件名`             | 当前组用于人工判断的代表帧                  | 可写多个，用逗号分隔                        | `01579.jpg,01689.jpg` |
| `负责人`                | 谁负责这个组                         | 写组员姓名                             | `张三`                  |
| `未开始/进行中/已确认`        | 当前组处理状态                        | 只填固定值                             | `进行中`                 |
| `备注`                 | 其他补充说明                         | 写无法归类、疑点、下一步动作                    | `疑似还混有远景视角`           |


### 补充：`视角编号` / 方位怎么判断

这里的“视角编号”在当前人工分组任务里，**不要理解成摄影学里的相机朝向编号**，更准确的理解是：

> **主作业区 / 计划 ROI / 主要目标簇，在画面横向上偏左、偏中还是偏右。**

建议按下面顺序判断：

1. 先看后续准备画 ROI 的主作业区；
2. 再看 person 最常出现、最密集的工作区域；
3. 最后再看整张图的横向重心是不是明显偏左或偏右。

可以直接用“三等分法”快速判断：

```text
| 左 1/3 | 中 1/3 | 右 1/3 |
```

- 主作业区明显落在左边，写 `左侧`；
- 主作业区主要落在中间，写 `居中`；
- 主作业区明显落在右边，写 `右侧`。

如果图片像很多加油站监控那样：

- 作业区基本覆盖整张图；
- 左右两边都有有效工作区域；
- 只是整体重心仍在中间附近；

那么**不要硬拆成左侧或右侧**，默认更推荐写：

- `居中`

并在 `备注` 里补一句：

- `工作区宽覆盖`
- 或 `全幅作业区`

也就是说，像这类“加油站工作区基本铺满画面”的图，当前更稳的口径不是“左/右机位”，而是：

> **方位写 `居中`，备注写 `工作区宽覆盖`。**

只有当主作业区明显只集中在左半边或右半边，并且这种差异会影响 ROI 共用时，才建议写成 `左侧` 或 `右侧`，甚至把它作为继续细拆的依据。

### 5.1 什么情况下才建议“继续细拆”

建议按下面这条判断链来做，而不是看到一点差异就立刻拆组：

```text
先看代表帧
-> 发现这个自动组里可能混了不同小场景 / 不同 ROI 边界
-> 再回看组内若干补充样本或边界样本
-> 确认这种差异是稳定存在的
-> 并且这种差异会影响“能不能共用一版 ROI”
-> 这时才把它正式细拆成多个人工最终组
```

更具体地说，下面几类情况通常可以作为“继续细拆”的正式依据：

1. **白天 / 夜晚稳定分成两批**
  - 不是偶尔亮一点、暗一点；
  - 而是整体曝光、灯光、背景可见区域已经明显不同；
  - 导致 ROI 边界在两批图里不适合共用一版。
2. **近景 / 中景 / 远景稳定分成两批或三批**
  - 不是单张图偶尔人更大；
  - 而是同组里长期混有“明显近景”和“明显远景”；
  - 这会导致作业区占画面的比例、有效 ROI 区域、下游 personcrop 价值都不一样。
3. **左侧 / 居中 / 右侧 或不同机位方向稳定混杂**
  - 不是构图轻微偏一点；
  - 而是作业区明显处在不同位置，ROI polygon 不能稳定复用；
  - 这时更适合按视角方位继续拆。
4. **同一自动组里其实混了两个以上作业区 / 场景区域**
  - 背景建筑、油机、地面线条、遮挡关系明显不同；
  - 即使分辨率一致，也不应强行共用一个组级 ROI。
5. **同一自动组里混有少量异类图，但异类已经形成稳定小簇**
  - 这类图不是 1~2 张偶发异常；
  - 而是可以稳定归成一个小子组；
  - 这时建议正式拆成子组，而不是只在备注里写一句“好像不一样”。

反过来，如果只是下面这种情况，通常**先不要拆整个组**：

- 只有极少数 `5%` 左右的异类图；
- 差异只体现在个别人更近一点、更远一点，但 ROI 边界本身仍基本可共用；
- 只是个别图片存在疑似标注问题。

这类更适合：

- 在 `图片明细` 里单独记录；
- 标记成 `逐图ROI`、`待定` 或 `疑似标注问题`；
- 而不是把整个自动组全部拆散。

### 5.2 `组级台账` 怎么具体填写

 建议组员按下面顺序逐列填写：

1. 先确认自己负责哪个自动组；
2. 先看 `group_summary.json` 里的代表帧；
3. 填：
  - `人工组名称/场景名`
  - `摄像头编号`
  - `场景编号/区域名`
  - `视角编号`
4. 再判断并填：
  - `稳定性等级`
  - `是否可共用一版ROI`
  - `是否需要继续细拆`
  - `组级ROI/逐图ROI/暂缓`
5. 最后填：
  - `负责人`
  - `状态`
  - `备注`

### 5.3 如果一个自动组要拆成多个人工组，怎么填

不建议直接把原自动组那一行覆盖掉。更稳妥的方式是：

1. **保留母组行**，让大家知道这批图最初来自哪个自动组；
2. **在母组行下面新增若干子组行**；
3. 所有子组行的 `当前自动组ID（来自t006）` 仍然写原始自动组，例如都写 `group_0001`；
4. 再用 `人工细分后的最终组ID` 去区分真正的人工子组。

也就是说，如果 `group_0001` 需要继续细拆，**可以，而且建议**写成：

- `group_0001_a_day`
- `group_0001_b_night`
- `group_0001_c_far`

这种形式。

推荐命名规则：

```text
<原自动组ID>_<字母序号>_<主要细拆依据>
```

例如：

- `group_0001_a_day`
- `group_0001_b_night`
- `group_0001_c_midfar`
- `group_0001_d_center`

建议：

- `ID` 尽量用英文 / 数字，便于后续目录和脚本使用；
- 中文解释写在 `人工组名称/场景名`；
- `ID` 里只放**最主要**的拆分依据，不要把所有信息都堆进 ID。

例如：

- `人工细分后的最终组ID`：`group_0001_a_day`
- `人工组名称/场景名`：`白天中景居中视角`

这样既便于机器处理，也便于人看懂。

#### 5.3.1 母组行和子组行分别怎么写

下面给一个更具体的填写示例。

假设 `group_0001` 原来有 `320` 张图，后来发现它实际上应拆成“白天”和“夜晚”两组。

那么 `组级台账` 建议这样写：


| 当前自动组ID（来自t006） | 当前组图片数 | 人工细分后的最终组ID        | 稳定性等级：A/B/C | 组级ROI/逐图ROI/暂缓 | 是否可共用一版ROI | 是否需要继续细拆 | 继续细拆依据       | 负责人 | 状态  | 备注                       |
| --------------- | ------ | ------------------ | ----------- | -------------- | ---------- | -------- | ------------ | --- | --- | ------------------------ |
| group_0001      | 320    | group_0001         | B           | 暂缓             | 否          | 是        | 白天/夜晚        | 张三  | 进行中 | 母组参考，准备拆成 day/night 两个子组 |
| group_0001      | 210    | group_0001_a_day   | A           | 组级ROI          | 是          | 否        | 白天           | 张三  | 进行中 | 由 group_0001 细拆而来        |
| group_0001      | 110    | group_0001_b_night | B           | 暂缓             | 待定         | 是        | 夜晚内部仍需继续看远近景 | 张三  | 进行中 | 由 group_0001 细拆而来        |


这个写法的好处是：

- 一眼就能看出 `group_0001_a_day` 和 `group_0001_b_night` 都来自 `group_0001`；
- 母组行还在，队友不会丢失来源；
- 子组行才是真正后续要拿去做 ROI 判断和落目录的对象。

#### 5.3.2 子组行里哪些字段最关键

一旦开始细拆，下面几个字段一定要认真填：

- `当前自动组ID（来自t006）`
- `人工细分后的最终组ID`
- `当前组图片数`
- `稳定性等级：A/B/C`
- `组级ROI/逐图ROI/暂缓`
- `是否可共用一版ROI`
- `是否需要继续细拆`
- `继续细拆依据`
- `备注`

其中最容易漏的是：

- **子组图片数**：不要一直沿用母组图片数；
- **是否需要继续细拆**：子组也可能还需要再拆；
- **备注**：最好写清“由哪个母组拆来”“当前还剩什么问题”。

## 6. `图片明细` 工作表：每一个标题的意义

 这个表不是让你把 2507 张图全填进去，而是只记录：

- 需要继续细拆的图片；
- 需要逐图 ROI 的图片；
- 疑似标注问题图片；
- 难以归类的图片。


| 标题               | 意义                     | 怎么填                                                   | 示例                 |
| ---------------- | ---------------------- | ----------------------------------------------------- | ------------------ |
| `图片stem（不含扩展名）`  | 图片唯一 stem              | 直接写文件名去掉扩展名                                           | `00231`            |
| `图片文件名`          | 图片完整文件名                | 带扩展名                                                  | `00231.jpg`        |
| `自动粗分组ID`        | 当前图片最初属于哪个自动组          | 参考 `group_manifest.csv`                               | `group_0002`       |
| `人工最终组ID`        | 这张图最终应归到哪个人工组          | 还未确定可先空或写待定                                           | `group_0002_a_day` |
| `摄像头编号`          | 该图片判断出的摄像头             | 不确定可空                                                 | `cam01`            |
| `场景编号/区域名`       | 场景或区域标识                | 写便于队伍统一的名称                                            | `zoneA`            |
| `视角编号`           | 视角标识                   | 保持和主表一致                                               | `view1`            |
| `白天/夜晚`          | 昼夜属性                   | 统一填 `白天 / 夜晚 / 待定`                                    | `白天`               |
| `近景/远景`          | 距离属性                   | 统一填 `近景 / 远景 / 中景 / 待定`                               | `远景`               |
| `左偏/右偏/居中`       | 主作业区在单张图中的横向位置属性       | 统一填 `左偏 / 右偏 / 居中 / 待定`；如果工作区基本铺满整张图，优先写 `居中（工作区宽覆盖）` | `居中`               |
| `组级ROI/逐图ROI/暂缓` | 这张图当前的 ROI 策略          | 只填固定值                                                 | `逐图ROI`            |
| `是否疑似标注问题`       | 这张图是否怀疑 person 标签本身有问题 | 建议填 `是 / 否 / 待定`                                      | `是`                |
| `保留/细拆/逐图ROI/暂缓` | 这张图下一步动作               | 只填固定值                                                 | `细拆`               |
| `备注`             | 其他说明                   | 写原因即可                                                 | `同组内视角差异明显`        |


### 6.1 `图片明细` 什么时候必须填

 遇到下面情况时建议一定要填：

1. 你发现一张图明显不该留在当前组；
2. 你发现某张图需要逐图 ROI；
3. 你怀疑这张图的 person 框本身有问题；
4. 你准备把一个大组继续拆开，需要记录“拆分依据样本”。

### 6.2 `图片明细` 怎么填最省时间

 建议不要先录很多普通图，只录：

- 典型边界样本；
- 典型异类样本；
- 需要证明“为什么拆组”的样本；
- 需要证明“为什么逐图 ROI”的样本。

## 7. `ROI标注计划` 工作表：每一个标题的意义

 这个表用于：当某个组已经基本确认后，再安排实际 ROI 标注任务。


| 标题                  | 意义                 | 怎么填                      | 示例                        |
| ------------------- | ------------------ | ------------------------ | ------------------------- |
| `人工最终组ID`           | 当前要标 ROI 的最终组 ID   | 与前面台账保持一致                | `group_0003`              |
| `人工组名称`             | 当前组的中文名称           | 和主表一致                    | `近景白天中间视角`                |
| `group / per_image` | ROI 范围类型           | 只填 `group` 或 `per_image` | `group`                   |
| `ROI工作目录`           | 后续 ROI 工作目录        | 建议写实际路径                  | `...\group_0003\roi-work` |
| `组内图片目录`            | 当前组图片所在目录          | 指向 `images/`             | `...\group_0003\images`   |
| `组内标签目录`            | 当前组标签所在目录          | 指向 `labels/`             | `...\group_0003\labels`   |
| `ROI JSON 输出目录`     | Labelme JSON 输出目录  | 后续建议单独建目录                | `...\group_0003\roi-json` |
| `建议代表帧数量`           | 如果是组级 ROI，建议抽多少代表帧 | 一般填 `5~10`               | `8`                       |
| `建议抽检数量`            | 画完 ROI 后建议再抽查多少图   | 一般填 `10~20`              | `15`                      |
| `未开始/进行中/已完成`       | 当前 ROI 标注状态        | 只填固定值                    | `未开始`                     |
| `标注负责人`             | 负责画 ROI 的人         | 写姓名                      | `李四`                      |
| `计划完成日期`            | 计划完成时间             | 用统一日期格式                  | `2026-05-20`              |
| `备注`                | 其他说明               | 如“本组先保守宽ROI”             | `先做组级ROI`                 |


### 7.1 `ROI标注计划` 什么时候开始填

 只有当一个组已经在 `组级台账` 中被基本确认后，才建议开始填这个表。

 不要在分组还没稳定时，就提前给它排 ROI 标注计划，否则很容易返工。

## 8. 组员具体需要怎么做：详细操作步骤

 下面给出一个尽可能详细的工作流程，适合直接发给队友照着做。

### Step 0：先看说明，不要先动图片

 每位组员先打开：

1. `new_person_labels_人工分组记录表模板.xlsx`
2. `new_person_labels_frames逻辑分组.md`
3. `new_person_labels_roi_grouping_t006/group_summary.json`

 先理解：

- 当前主参考版本是 `t006`；
- 当前不是直接开始 ROI 标注；
- 而是先把自动组整理成“人工最终组”。

### Step 1：认领自动组

 组长先给每位组员分组，例如：

- 甲：`group_0002`
- 乙：`group_0005`
- 丙：`group_0006`
- 丁：`group_0001 + group_0003 + group_0004`

 组员在 `组级台账` 里先把自己负责的组对应行找出来。

### Step 2：看代表帧，先做第一轮判断

 每位组员先根据 `group_summary.json` 里的 `representatives`，打开这些代表帧去看：

- 背景像不像同一场景；
- 视角像不像同一机位；
- 作业区边界大致一不一致；
- 是不是明显混了白天 / 夜晚、远景 / 近景。

 这一轮不要急着填很细的图片明细，先做组级判断。

### Step 3：先填写 `组级台账` 的基础字段

 每位组员先填：

- `人工组名称/场景名`
- `摄像头编号（未知可留空）`
- `场景编号/区域名`
- `视角编号`
- `负责人`

 如果不知道，就不要乱编。可以先留空或写 `待定`。

### Step 4：判断稳定性和 ROI 策略

 然后再填：

- `稳定性等级：A/B/C`
- `是否可共用一版ROI`
- `是否需要继续细拆`
- `组级ROI/逐图ROI/暂缓`

 推荐判断口径：

- **A**：可共用一版 ROI，优先组级 ROI；
- **B**：大致同场景，但仍要继续拆；
- **C**：明显不稳定，逐图 ROI 或暂缓。

### Step 5：如果要继续细拆，就先写“拆分依据”

如果你判断这个组还需要拆，就在：

- `是否需要继续细拆`
- `继续细拆依据（如白天/夜晚/远近景）`

里先写清楚。

这里要特别注意：

> **“细拆依据”不是随便写一个感觉词，而是要写“为什么这批图已经不适合继续共用一版 ROI”的主要原因。**

推荐顺序是：

```text
先看代表帧
-> 再抽几张组内补充图确认这种差异不是偶发
-> 再决定按什么维度拆
-> 先在台账里保留母组行
-> 再新增子组行
-> 最后再落到子组目录
```

例如：

- `是`
- `白天/夜晚`

或：

- `是`
- `远景/近景`

再例如，如果你确认 `group_0001` 要拆成两组，推荐这样处理：

1. 在 `组级台账` 里保留母组行：
  - `当前自动组ID（来自t006）= group_0001`
  - `人工细分后的最终组ID = group_0001`
  - `是否需要继续细拆 = 是`
  - `继续细拆依据 = 白天/夜晚`
  - `组级ROI/逐图ROI/暂缓 = 暂缓`
2. 然后在它下面新增子组行：
  - `group_0001_a_day`
  - `group_0001_b_night`
3. 再分别填写每个子组自己的：
  - 图片数
  - 稳定性等级
  - ROI 策略
  - 是否还要继续拆

### Step 5.1：细拆后的目录结构怎么建

当前更推荐的做法，不是把细拆后的子组直接平铺到 `grouped/` 根目录，而是**保留母组目录作为来源工作区**，再在母组目录下面建子组目录。

如果 `group_0001` 需要细拆，推荐目录结构写成：

```text
backend-train-model/person-train-model/train-result/working/
└─ new_person_labels_frames_grouping_t006/
   └─ grouped/
      └─ group_0001/
         ├─ images/                    # 母组原始图片池 / 参考池
         ├─ labels/                    # 母组原始标签池 / 参考池
         └─ manual_split/
            ├─ group_0001_a_day/
            │  ├─ images/
            │  └─ labels/
            ├─ group_0001_b_night/
            │  ├─ images/
            │  └─ labels/
            └─ undecided/
               ├─ images/
               └─ labels/
```

这里各层的含义是：

- `group_0001/images`、`group_0001/labels`
  - 保留原自动组母组工作区；
  - 不建议因为细拆就马上删掉。
- `manual_split/`
  - 表示这是人工继续细拆出来的子组目录。
- `group_0001_a_day/`、`group_0001_b_night/`
  - 表示真正的人工最终组候选目录。
- `undecided/`
  - 用来临时放还没想清楚该归哪一组的边界样本；
  - 这样不会硬塞进错误子组里。

如果后续 `group_0001_b_night` 里面还要继续拆，也继续在它下面按同样思路处理，但**不建议无限嵌套很多层**。一般超过两层，说明前面的拆分逻辑还需要重新整理。

#### Step 5.2：目录和表格要保持一一对应

只要你新建了一个子组目录，就建议在 `组级台账` 中同步新增对应子组行。

例如你建了：

- `group_0001_a_day/`
- `group_0001_b_night/`

那么表格里也应同步有：

- `人工细分后的最终组ID = group_0001_a_day`
- `人工细分后的最终组ID = group_0001_b_night`

不要出现下面这种情况：

- 目录里已经拆了 3 个子组；
- Excel 里还是只有一行 `group_0001`。

否则后面谁都不知道目录和台账该怎么对应。

### Step 6：再把关键图片补到 `图片明细`

 当某组已经判断出需要细拆或逐图 ROI 后，再去 `图片明细` 里记录：

- 哪些图片是典型样本；
- 哪些图片不该和主组混在一起；
- 哪些图片疑似标注问题；
- 哪些图片必须逐图 ROI。

 不建议把组内所有图片都抄进去，优先记录：

- 边界图；
- 异类图；
- 拆分证据图；
- 标注问题图。

### Step 7：当某个人工组稳定后，再填 `ROI标注计划`

 当且仅当你已经能比较确定：

- 这是一组稳定图；
- 大多数图可以共用一版 ROI；

 才去填：

- `ROI标注计划`

 这时再安排：

- 代表帧抽多少张；
- 由谁画；
- 抽检多少张；
- 何时完成。

### Step 8：组长最后统一收口

 组长最后要检查三件事：

1. 大组是否真的被拆开；
2. ROI 策略字段是否统一使用固定口径；
3. 进入 ROI 标注计划的组，是否已经足够稳定。

## 9. 常见错误

### 错误 1：直接把自动组当最终组

 自动组只是第一轮候选组，不等于最终人工确认后的 ROI 组。

### 错误 2：一上来就逐图 ROI

 当前更推荐：

- 先组级判断；
- 只有少量不稳定图再逐图 ROI。

### 错误 3：在 Excel 里乱写自由口径

 尤其这些字段必须统一：

- 稳定性等级：`A/B/C`
- ROI 策略：`组级ROI/逐图ROI/暂缓`
- 状态：`未开始/进行中/已确认` 或 `未开始/进行中/已完成`

### 错误 4：还没稳定就先排 ROI 标注计划

 先分组，后标 ROI。不要反过来。

## 10. 一句话执行口径

> **组员的工作不是“把 Excel 填满”，而是基于 `t006` 自动粗分组结果，先把自动组整理成可以稳定共用 ROI 的人工最终组：先填组级台账，再用图片明细记录边界样本和拆分依据，最后只对确认稳定的组填写 ROI 标注计划。**
