# 新增样本训练接入说明

本文档用于回答当前这类实际问题：

- `person` 要继续加样本时，是否需要改训练策略；
- `clothes` 要继续加样本时，如果新样本里的工服款式和当前训练集不一样，应该在标注阶段拆类，还是先保持单类，后面再处理。

本文档基于仓库当前事实给出建议，而不是泛泛讨论“目标检测通常怎么做”。当前仓库里：

- `person` 仍是单类：`0 -> person`
- `clothes` 仍是单类：`0 -> clothes`
- 在线工服链路默认把命中 `clothes` 视为“穿了合规工服”的候选证据，相关配置在 `inspection-flask/settings.py`

因此，**是否拆类**不能只看标注方便不方便，还必须一起看训练配置、评估口径和在线规则。

## 1. 当前默认前提

### 1.1 `person`

- 当前 `person` 任务语义很稳定，就是“检测人”。
- 只要新增样本没有引入新的任务定义，通常不需要改成多类，也不需要围绕“类别设计”做额外文章。
- 真正需要注意的是：
  - 是否有新的机位 / 新的背景域；
  - 是否有新的 ROI 区域；
  - 是否会破坏当前按序列切分的评估口径。

### 1.2 `clothes`

- 当前 `clothes` 训练与在线链路默认都按**单类工服检测**工作。
- 现有训练配置中的 `class_names` 默认是：

```json
{
  "0": "clothes"
}
```

- 在线链路当前默认配置是：
  - `WORKWEAR_LABELS = ["clothes"]`
  - `WORKWEAR_COMPLIANCE_MODE = "any"`

这意味着：**只要模型检测到 `clothes`，当前在线逻辑就会把它当作“合规工服命中”来参与后续判断。**

所以，如果你把“新款工服”直接拆成新类，这不只是数据标注变化，还会影响：

- 训练集 `class_names`
- 评估结果的解释方式
- 在线 `WORKWEAR_LABELS`
- 在线合规判定模式 `any / all`

## 2. `person` 新增样本怎么处理

当前建议很直接：**正常加样本、正常训练，不把重点放在类别设计上。**

### 2.1 标注建议

- 继续保持单类 `person`。
- 保持和现有数据一致的框语义，不要这批框全身、下一批只框半身，导致标签风格漂移。
- 如果新样本来自新机位或新视频序列，仍然按“序列”为单位组织，不要把同一段连续视频的相邻帧随意打散到 `train / val / test`。

### 2.2 数据准备建议

- 如果只是 fullframe person 扩样，优先沿用现有 `person_fullframe` 准备和训练流程。
- 如果这些新样本后续也要进入 ROI-aware person 分支，就必须同步补齐 ROI 标注 / ROI JSON，而不是只补 `person` 框。
- 如果新增样本主要来自新的场景域，建议单独留一部分做显式 holdout，用来判断是不是“样本加了很多，但新场景还是不稳”。

### 2.3 训练策略建议

- 第一轮不要急着改 `imgsz`、模型尺寸或 keep rule。
- 先按当前稳定主线继续：
  - fullframe `person`
  - ROI-aware `person`
- 先看新增样本后，当前主线指标是否稳定，再决定是否需要进一步改 ROI 规则或训练参数。

## 3. `clothes` 新增样本怎么处理

这里才是这次需要重点处理的部分。

### 3.1 默认推荐：**先不要因为“款式不同”就拆类别**

如果新样本中的工服只是：

- 颜色不同；
- 款式不同；
- 反光条样式不同；
- 新旧批次制服外观不同；

但业务语义仍然都是“这算合规工服”，那么当前更推荐：

**标注时仍统一标成 `0 clothes`，不要先拆成 `clothes_old / clothes_new`。**

原因很明确：

1. 当前仓库的训练和在线链路都默认 `clothes` 是单类。
2. 你现在面对的核心问题更像是**domain shift / appearance shift**，不是任务定义变了。
3. 如果仅因为外观差异就拆类，模型学习目标会从“识别工服”变成“先区分工服款式”，这未必对当前业务最有利。
4. 一旦拆类，后续评估和在线规则都要同步调整，工作量和风险都会上升。

### 3.2 更好的做法：**类别不拆，样本元数据单独记录**

推荐把“新旧工服差异”记录为样本元数据，而不是直接写进 `class_id`。

可以记录的信息包括：

- `source_id`
- `sequence_name`
- `style_tag`
- `camera_id`
- `site_id`
- `remark`

其中 `style_tag` 可以类似：

- `old_uniform`
- `new_uniform`
- `mixed_uniform`

这样做的好处是：

- 当前单类训练可以直接兼容；
- 后面如果要做“按款式切分评估”，不需要返工重标框；
- 后面如果要做“多类对照实验”，也可以在已有框基础上重映射类别，而不是整批返工。

### 3.3 标注时真正要守住的是“框语义一致”

对 `clothes` 来说，最怕的不是新款式，而是**新旧数据的框语义不一致**。

例如下面这些情况要避免：

- 旧数据框的是整块工服区域，新数据只框胸前一小块标识；
- 旧数据框的是可见工服主体，新数据只框反光条；
- 旧数据遇到遮挡仍保守标注，新数据遇到遮挡直接不标；
- 旧数据偏宽松，新数据偏紧，导致模型学到两套不一致的框风格。

因此，新增 `clothes` 样本时，优先做的是：

- 先拿几张旧样本和几张新样本并排核对；
- 确认框的语义一致，再批量标注。

## 4. 什么时候才建议把 `clothes` 拆类

不是永远不能拆，而是**不要把“款式不同”自动等同于“应该拆类”**。

### 4.1 可以考虑拆类的场景

只有在下面这些情况下，拆类才更合理：

1. 业务上确实要区分不同工服类型。
2. 不同工服类型对应不同告警规则。
3. 你们要做的不是“是否穿工服”，而是“是否同时满足多个部件要求”。

### 4.2 不建议按“旧款 / 新款”直接拆类的原因

如果业务上旧款和新款都算“合规工服”，那把它们拆成两个检测类，通常只是把问题复杂化：

- 训练阶段要学两个外观分布；
- 推理阶段还要把两个类重新映射回“都算合规”；
- 评估时总分和分类分数都会变得更难解释；
- 新样本数量如果不均衡，还可能带来更明显的类别不平衡。

### 4.3 如果未来真的要拆，优先拆“规则语义类”，不是“款式类”

如果业务将来变成：

- 上衣必须有；
- 反光背心必须有；
- 其他防护部件必须有；

那更合理的拆法是按**部件 / 规则语义**拆类，而不是按 `old/new style` 拆类。

因为在线链路已经预留了：

- `WORKWEAR_COMPLIANCE_MODE = "any"`
- `WORKWEAR_COMPLIANCE_MODE = "all"`

这更适合承接“命中任一类算合规”或“必须多个部件同时命中才算合规”的规则，而不是承接“只是款式不同”的外观差异。

## 5. 新增 `clothes` 样本时的推荐实验顺序

当前最稳妥的顺序不是“先拆类再说”，而是下面这样：

### 第一步：单类 mixed baseline

- 新旧工服都标成 `clothes`
- 合并进入训练集
- 保持当前主线训练口径

先回答最关键的问题：

**“只做单类混训，模型能不能已经学会新款工服？”**

### 第二步：单独保留新款工服 holdout

不要把所有新款工服都塞进训练集。

至少要单独留出一部分新款工服样本，用来回答：

- 总指标是不是还不错；
- 但模型是不是其实只会旧工服，不会新工服。

如果没有这一步，你最后只会得到一个“总体分数”，但不知道模型对新款式到底稳不稳。

### 第三步：如果新款 holdout 仍然差，先补样本，再谈拆类

如果 mixed 单类训练后，新款工服表现仍明显偏差，优先考虑：

- 继续补新款工服样本；
- 补足不同姿态、距离、遮挡、光照下的新款样本；
- 检查新旧样本框语义是否一致；
- 检查 split 是否让新款工服过度集中在训练集或测试集某一侧。

在这一步之前，不建议默认把问题归因为“必须拆类”。

### 第四步：最后才做多类对照实验

只有当前三步都做完，仍然表明单类 mixed 学不稳，才建议做多类对照：

- `clothes_old`
- `clothes_new`

但这应该是**对照实验**，不是第一反应。

## 6. 如果真的做多类，需要同步改哪些地方

如果未来确实决定把 `clothes` 拆类，不能只改标注文件。

至少要同步检查这些入口：

### 6.1 训练配置

- `backend-train-model/project_config.json`
- `backend-train-model/All-train-model/*.build.json`

里面的 `class_names` 需要从：

```json
{
  "0": "clothes"
}
```

改成新的多类映射。

### 6.2 数据构建与评估口径

- merged 数据集构建后，新的 `dataset.yaml` 会带上新的类别名；
- 旧的单类 baseline 指标和新的多类指标，解释方式不再完全一样；
- 原来的统一 holdout 如果还要继续做“主线对照”，最好保持一份单类口径不动，避免基线混乱。

### 6.3 在线规则

如果旧款和新款都算合规工服，那么在线至少要同步修改：

- `inspection-flask/settings.py`

例如：

- `WORKWEAR_LABELS = ["clothes_old", "clothes_new"]`
- `WORKWEAR_COMPLIANCE_MODE = "any"`

如果未来变成“必须同时命中多个部件类”，那才可能改成：

- `WORKWEAR_COMPLIANCE_MODE = "all"`
- `WORKWEAR_REQUIRED_LABELS = [...]`

## 7. 对当前仓库更稳妥的接入建议

结合当前仓库状态，新增样本时更推荐下面这版流程。

### 7.1 `person`

- 继续单类 `person`
- 正常补样本
- 继续按序列切分
- 如果新样本会进入 ROI-aware 分支，同步补 ROI 配置

### 7.2 `clothes`

- 继续单类 `clothes`
- 新旧工服先统一标成 `clothes`
- 在样本清单或 manifest 里补 `style_tag`
- 至少留一部分新工服做独立 holdout

### 7.3 评估

新增样本后，至少同时看两类结果：

1. 总体指标是否变化；
2. 新工服子集是否稳定。

如果只看总体分数，不足以说明“新款工服已经被学会”。

## 8. 当前建议的明确结论

当前更推荐的结论是：

1. `person` 不需要围绕类别设计做重点改动，正常补样本、正常训练即可。
2. `clothes` 如果只是工服款式不同，但业务语义仍然都是“合规工服”，**不要在第一步就按款式拆类**。
3. `clothes` 更好的做法是：
   - 标注仍统一为 `clothes`
   - 额外记录 `style_tag / source_id / sequence_name`
   - 单独保留新款工服 holdout
4. 只有当业务规则真的不同，或者单类 mixed 实验证明始终学不稳时，才把拆类升级为正式方案。

这份建议的核心不是“永远单类”，而是：

**先把业务语义、训练口径和在线规则保持一致，再用对照实验决定是否值得把问题升级为多类检测。**
