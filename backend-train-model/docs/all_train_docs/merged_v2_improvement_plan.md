# `merged_v2` 提升方案与实验优先级

本文档专门回答 4 个问题：

1. 面向后续接入摄像头、走真实生产时，三套数据是继续分开训练更合适，还是整合训练更合适；
2. 训练参数是否会显著影响模型效果；
3. 当前 `merged_v2` 为什么没有明显优于 `first-train`；
4. 接下来应该优先改什么，才能让 merged 路线更有机会真正跑赢基线。

---

## 1. 先给结论

### 1.1 生产目标上，更推荐“一个 merged 主模型 + 分源评估”

如果后续目标是：

- 接入真实摄像头；
- 覆盖不同机位、时段、背景、人员距离；
- 最终接到 `inspection-flask` 的在线链路；

那么长期更推荐的方向不是维护 `3` 个彼此独立的最终生产模型，而是：

- **保留分开训练结果作为诊断基线；**
- **把 merged 模型作为最终主线候选。**

原因很直接：

- 摄像头上线后，线上输入天然是混合域，不会严格只来自单一数据源；
- 如果继续维护多套独立模型，后续部署、切换、回归验证、版本管理成本都会上升；
- merged 模型更符合“为真实生产铺垫”的方向。

但这不等于：

- “只要把三套数据机械拼起来训练，就一定比单套更好”。

当前仓库里的真实结果已经说明：

- **merged 方向是对的；**
- **当前这版 `merged_v2` 还没有把这条方向跑通。**

### 1.2 参数会影响效果，但当前主要矛盾不是参数

训练参数当然会影响效果，尤其是：

- 初始化权重；
- 学习率与训练时长；
- 输入尺寸；
- batch size；
- 数据增强；
- 正负样本比例。

但结合当前仓库已有结果，更大的矛盾仍然是：

1. 数据集构造方式；
2. `train / val / test` 的切分方式；
3. review 空标签样本的引入方式；
4. merged 后不同来源之间的分布不均衡。

换句话说：

- **参数重要；**
- **但当前 `merged_v2` 和 `first-train` 拉开明显差距的主因，更像数据与评估口径，而不是单纯超参数没调好。**

---

## 2. 关于当前 `merged_v2_from_first` 的一个重要说明

需要明确记录一个事实，避免后续误判：

- 当前 `clothes_merged_v2_from_first` 这条路线，**最初启动时确实是用 `first-train` 的 `best.pt` 作为初始化权重**；
- 后来因为训练过程中断，才继续使用同一条 run 的 `last.pt` 做严格断点续训；
- 因此，**不能把当前结果落后简单归因于“没有从 `first-train` 的 `best.pt` 起跑”**。

当前仓库里保留下来的训练报告之所以显示为 `resume`，是因为它记录的是：

- 中断后继续训练的这一段；

而不是最初启动 `merged_v2_from_first` 的那一次完整起跑信息。

因此更准确的判断应该是：

- **本轮结果差，不是因为起跑权重选错了；**
- **更可能是 merged 数据构造、split 口径、review 空标签引入方式等问题，把模型拖弱了。**

---

## 3. 当前证据显示了什么

### 3.1 `first-train` 仍然是当前更稳的基线

按当前仓库内现有评估结果看：

- `first-train` 的 `val` 指标明显强于 `merged_v2_from_first`；
- `merged_v2_from_first` 只是在 `test precision` 上更高，但整体并没有形成稳定领先。

更关键的是训练稳定性：

- `first-train` 的最佳结果更高；
- 而且训练后段没有明显崩掉；
- 当前 `merged_v2_from_first` 则存在中后段退化较明显的问题。

因此到目前为止，更适合继续保留为基线的是：

- `first-train/artifacts/runs/clothes_fullframe_baseline/weights/best.pt`

### 3.2 `merged_v2` 增加的主要不是正样本，而是补齐后的空标签样本

从当前两版 merged 数据集构造结果看：

#### `merged_v1_positive_only`

- `included_images = 454`
- `included_boxes = 980`
- `pending_review_items = 48`
- split 为：
  - `train = 330`
  - `val = 62`
  - `test = 62`

#### `merged_v2_full_reviewed`

- `included_images = 502`
- `included_boxes = 980`
- `pending_review_items = 0`
- `resolved_review_items = 48`
- split 为：
  - `train = 357`
  - `val = 83`
  - `test = 62`

这里最值得注意的是：

- 图片数增加了 `48`；
- 但标注框总数仍然是 `980`，没有增加。

这说明：

- `merged_v2` 相比 `merged_v1`，新增的核心内容不是更多正样本框；
- 而是此前缺标、后来补成 review 空标签的样本。

这类样本不是不能加，恰恰相反，它们对真实生产非常重要；
但如果引入方式不对，会直接改变训练和验证分布。

### 3.3 当前 `val` 与 `test` 的来源结构并不均衡

当前 merged 构造中：

- `val` 主要压在 `g33 / D02_20260123070624`；
- `test` 主要来自 `g31 / D04_20260123074846` 与 `g31 / D15_20260123074848`；

这会带来一个非常现实的问题：

- 你看到的“`val` 很差、`test` 还行”，不一定等于“模型真的泛化得很好”；
- 它也可能只是说明：
  - `val` 更难；
  - `test` 更容易；
  - 或者 `val / test` 的数据风格本来就不一致。

所以当前结果更适合解读为：

- **merged 模型还没有在统一、可信的比较口径下证明自己优于 `first-train`。**

## 4. 为什么 `merged_v2` 现在会明显弱于 `first-train`

我更倾向于按下面的优先级理解根因。

### 4.1 第一层原因：比较口径本身还不够干净

当前 `first-train` 与 `merged_v2_from_first`：

- 数据规模不同；
- 来源结构不同；
- holdout 口径不同；
- `val / test` 的域分布也不同。

所以现在的结论应该是：

- `merged_v2_from_first` **没有证明自己比 `first-train` 更强**；
- 但还不能把所有差距都简单理解为“模型结构或参数一定不行”。

这是当前第一优先级的比较边界。

### 4.2 第二层原因：review 空标签样本改变了验证集难度

`merged_v2` 比 `merged_v1` 多出来的 `48` 张样本，主要是 review 后确认的空标签图。

其中最敏感的是：

- `val` 从 `62` 张变成 `83` 张；
- 但 `val` 的标注框总数仍然是 `74`。

这意味着：

- `val` 集里“没有 `clothes` 框的图”占比提高了；
- 如果这些空标签图里存在：
  - 边缘可见工服；
  - 模糊但仍可判为 `clothes` 的目标；
  - 原本就应当有框、但被误写为空标签的情况；

那么模型在这些图上的预测会被直接计作 false positive。

这会非常明显地拖低：

- precision；
- recall；
- `mAP50`；
- `mAP50-95`。

所以这 48 张 review 图不是“加了就天然更接近生产”，而是必须分清：

1. 真负样本；
2. 难负样本；
3. 疑似漏标样本；
4. 标注边界不清晰样本。

### 4.3 第三层原因：merged 后的多源分布并没有被平衡使用

当前 merged 数据来自多组序列，来源之间在以下方面可能不同：

- 背景；
- 拍摄角度；
- 目标大小；
- 光照；
- 工服外观；
- 人员密度；
- 画面中干扰物。

如果只是简单按“当前文件存在与否”来纳入 merged，而没有进一步处理：

- 每个来源的占比；
- 每个来源在 `train / val / test` 中的代表性；
- 正负样本比例；

那 merged 模型往往会出现：

- 在某些来源上更稳；
- 在另一些来源上明显退化；
- 总体看起来不稳定。

这和当前结果里出现的：

- `val` 很弱；
- `test` 还不算完全崩；

是相符的。

### 4.4 第四层原因：同一套默认训练参数未必适合更大的 merged 数据

当前几条主要训练结果大体都还沿用：

- `imgsz = 640`
- `epochs = 180`
- `batch = 4`
- `device = cpu`

这套参数对 `first-train` 可以工作，不代表对 merged 后更复杂的数据仍然是最好配置。

但这里要注意轻重缓急：

- **参数需要调；**
- **但不应先把主要精力放在参数上。**

如果数据、split、review 空标签质量没先理顺：

- 单纯改 `epochs`、`patience`、`imgsz`，大概率只能带来波动；
- 很难从根上解决当前差距。

### 4.5 断点续训本身不是当前主要问题

既然这条 run 的真实历史是：

- 先用 `first-train` 的 `best.pt` 启动；
- 中途中断；
- 再用同一 run 的 `last.pt` 继续；

那么这个 `resume` 逻辑本身是合理的。

因此当前不建议把主要问题归到：

- “resume 导致训练坏掉”。

更合理的说法是：

- `resume` 不是当前主要矛盾；
- 当前主要矛盾仍然是 merged 数据与评估口径。

当然，为了减少后续复盘歧义，后面最好把：

- “首次启动训练的记录”；
- “中断后 resume 的记录”；

在文档上显式写清楚，避免只看一份 train report 就误判起跑方式。

## 5. 参数到底该怎么看

### 5.1 参数确实会有较大影响

当数据与 split 稳定后，以下参数会显著影响效果：

- `--base-model` / 初始化权重；
- `imgsz`；
- `batch`；
- 学习率与调度；
- `epochs` 与 `patience`；
- 是否换更大模型；
- augmentation 强度。

### 5.2 但当前阶段的优先级不应把参数放第一位

对当前仓库来说，更推荐这样排序：

1. **先修数据与 split；**
2. **再做一版干净、可复现的 merged 训练；**
3. **最后再做参数扫描。**

如果顺序反过来，就容易出现：

- 实验很多；
- 指标也波动；
- 但无法确认到底是参数起作用，还是数据构造在变。

### 5.3 参数实验应该怎么做才有意义

推荐遵守两个原则：

#### 原则 A：一次只改一类因素

例如先固定：

- 数据集版本；
- holdout；
- 初始化权重；

然后只改：

- `imgsz`
- 或 `batch`
- 或模型尺寸

不要在同一轮里同时改：

- 数据集；
- split；
- 初始化权重；
- 训练参数；

否则结论会失真。

#### 原则 B：每次实验都要保留分源评估

即使最终上线只用一个 merged 主模型，也必须保留：

- 按 source 的对比；
- 按 sequence 的对比；

否则很容易出现：

- 总指标略有提升；
- 但某一路摄像头场景明显退化；

这种情况在真实生产里风险很高。

## 6. 推荐的推进路径

下面的顺序是我认为当前最值得执行的方案。

### `P0`：先固定比较口径

目标：

- 建立一套统一 holdout；
- 让 `first-train`、`merged_v1`、`merged_v2` 都在同一评估口径下比较。

完成标准：

- 有一套固定不再变化的 `val / test`；
- 三条训练路线都能在同一 holdout 上复评。

为什么优先做这个：

- 不先做这一步，后续所有“谁更好”的结论都带噪声。

### `P1`：把 48 张 review 空标签再分层复核一遍

目标：

- 不只是确认“有没有 txt”；
- 而是确认“空白 txt 是否真的合理”。

建议把这批 review 图按 4 类重新标注清单：

1. 明确真负样本；
2. 难负样本；
3. 疑似漏标；
4. 需要二次确认的边界样本。

完成标准：

- 对每张 review 图，都能说明为什么它是空标签；
- 不再把疑似漏标图混进空标签集合。

### `P2`：重做 source-balanced split

目标：

- 不让 `val` 主要只代表一个来源；
- 不让 `test` 主要只代表另一个来源。

建议原则：

- `train / val / test` 都尽量覆盖多 source；
- 每个 split 都要同时包含：
  - 易样本；
  - 中等难度样本；
  - 难样本；
- 空标签样本在 `train / val / test` 中的比例要受控。

完成标准：

- 不再出现“某个 split 基本只由某一组 source 构成”的情况；
- 不再出现 `val` 比 `test` 显著更偏某一类风格。

### `P3`：基于修正后的 merged 数据，重新跑一版干净训练

目标：

- 重新训练一版可以清晰解释的 merged 候选模型；
- 保证实验描述没有歧义。

推荐做法：

1. 显式记录初始化权重来自：
   - `first-train/.../best.pt`
2. 如果中途中断，可以继续 `resume`；
3. 但训练记录里要同时保留：
   - 首次启动信息；
   - 中断恢复信息；
4. 最终评估时，以完整 run 的结论为准。

注意：

- 如果只是同一条 run 中断后恢复，`resume` 本身不是问题；
- 关键是实验记录必须能追溯清楚。

### `P4`：在数据与 split 稳定后，再做参数实验

推荐参数实验顺序：

1. 保持模型不变，先试：
   - `imgsz`
   - `patience`
   - 学习率
2. 再考虑：
   - 更大模型；
   - 更强增强；
   - 更贴近线上使用方式的 prepare 流程。

如果前面三步没完成，不建议优先做大规模参数搜索。

## 7. 一个更适合生产的训练策略

当前更推荐的工程策略不是二选一，而是“三层结构”：

### 第 1 层：保留 `first-train` 作为稳定基线

作用：

- 保底；
- 对照；
- 发现 merged 是否真的有提升。

### 第 2 层：把 merged 当成主研发方向

作用：

- 逐步吸收多来源数据；
- 向真实摄像头混合域靠近；
- 为后续接入在线系统做铺垫。

### 第 3 层：保留分源评估，而不是回到分源独立生产模型

作用：

- 判断 merged 对每个来源是否都稳定；
- 识别哪个来源在拖后腿；
- 为后续站点微调提供依据。

所以更准确的建议不是：

- “只做分开训练”
- 或 “只做整合训练”

而是：

- **分开训练结果用于诊断与保底；**
- **整合训练结果用于冲击最终生产主模型。**

---

## 8. 下一轮最值得直接执行的动作

如果只允许选最重要的 5 件事，我建议按下面顺序做：

1. 固定统一 holdout，避免继续拿不同 split 做混合比较；
2. 复核 `48` 张 review 空标签，排掉疑似漏标；
3. 重做一版 source-balanced 的 merged 数据集；
4. 基于 `first-train` 的 `best.pt` 跑一版新的 merged 训练，并把首次启动与 resume 记录写清楚；
5. 在同一 holdout 下，再决定是否开始调 `imgsz`、模型尺寸、学习率等参数。

---

## 9. 本文档对应的判断边界

本文档明确不把以下内容当成已经成立的事实：

- 不把当前 `merged_v2` 写成“已经优于 `first-train`”；
- 不把“中断后 resume”写成当前主要失败原因；
- 不把“参数搜索”写成当前第一优先级；
- 不把“merged 方向正确”直接写成“当前 merged 结果已经可替代基线”。

当前更准确的表述应当是：

- **长期生产方向更偏向 merged；**
- **当前最稳的现成基线仍然是 `first-train`；**
- **`merged_v2` 需要先把数据、split、review 空标签质量和比较口径理顺，才有机会真正跑赢基线。**
