 # backend-train-model 后端模型训练总结（供加氢站工服检测项目参考）

 ## 1. 文档目的与适用范围

 本文档用于把当前仓库 `backend-train-model/` 下已经形成的后端模型训练体系，整理成一份可直接发给其他项目组的参考说明。

 重点覆盖四类内容：

 1. 当前各个 Python 脚本的作用和所处位置；
 2. 当前数据集切分方式、这样做的原因、以及更适合后续扩域的改进方向；
 3. 本项目在加油站场景下的后端工服检测思路，以及迁移到加氢站时的落地方案；
 4. 如果新成员要快速看懂代码结构，应该按什么顺序查看。

 需要先强调两个边界：

 - `backend-train-model/` 负责的是**训练、数据准备、评估、复盘和导出**；
 - 真正的在线告警链路还会依赖 `inspection-flask/` 中的 `person -> ROI -> workwear -> track -> temporal rule` 串联逻辑。

 因此，下面总结的是“后端训练体系”，不是“完整上线系统的全部代码”。

 ## 2. 当前后端训练体系总览

 ### 2.1 目录级分工

 当前 `backend-train-model/` 已经不是一条单脚本训练线，而是分成了四个相对清晰的子体系：

 | 子目录 / 模块 | 当前职责 |
 | --- | --- |
 | `backend-train-model/` 根目录 | 通用 clothes 训练底座，负责默认配置、通用数据准备、训练入口、多源切分 manifest 生成 |
 | `All-train-model/` | 多源 clothes 合并训练主线，包含当前固定 baseline、统一 holdout 与 baseline 复盘 |
 | `new_clothes_train/` | 新增工服样本接入专项，解决新 source 接入、空标注补齐、legacy split 继承问题 |
 | `person-train-model/` | person fullframe、ROI-aware person、ROI 标注衔接、误差复盘、人工复核导出 |

 ### 2.2 当前主线与关键基线

 当前最值得对外同步的，不是“所有历史实验”，而是下面几条已经比较明确的主线：

 | 训练线 | 当前状态 | 关键数据口径 | 当前结论 |
 | --- | --- | --- | --- |
 | `clothes merged baseline` | 当前最规范、最固定的 clothes 主基线 | `manifest + unified_holdout_v1` | 是当前 clothes 对外最稳的标准线 |
 | `clothes with new labels` | 新增工服样本接入后的可用基线 | 旧 7 源沿用 legacy manifest，新源 `gnew` 分层随机切分 | 适合扩样，但还不是最终跨域 holdout 终态 |
 | `person_fullframe_baseline` | 旧 502 图 person fullframe 基线 | `sequence_contiguous` | 是后续 ROI-aware 初始化来源 |
 | `person_fullframe_with_new_labels` | 新增 person 样本后的 fullframe 扩样主线 | `sequence_contiguous`，`roi.enabled=false` | 当前先做 fullframe 扩样，不把未补 ROI 的样本误接入 ROI-aware |
 | `person_roi_aware_v3_mask_then_crop_margin64_from_fullframe` | 当前 ROI-aware person 默认主线 | 基于旧 502 图，`mask_then_crop + margin64` | 相对历史 ROI-aware 有提升，但相对 v2 只是小幅优势，仍需 seed 稳定性确认 |

 ### 2.3 当前最值得记住的几个事实

 1. **clothes 主基线已经进入 manifest 驱动阶段**，不再只是默认 `0.7 / 0.15 / 0.15` 切分。
 2. **person 主线目前仍主要依赖 `sequence_contiguous`**，包括旧 502 图与扩样后的 fullframe 数据。
 3. **ROI-aware person 当前主要是在做 ROI 规则与 prepared 数据的迭代**，而不是已经形成跨域 holdout 的终局评测体系。
 4. **clothes 模型本身只是“工服证据检测子模型”**，不能直接等同于“未穿工服告警器”。
 5. **person + ROI 当前也只是筛 ROI 内候选作业人员**，不能直接写成“已完成工人身份识别”。

 ### 2.4 当前关键指标速览

 #### 2.4.1 clothes merged 当前主 baseline

 - 当前 run：`clothes_merged_v2_balanced_from_first_holdout_v1`
 - 统一对照集：`All-train-model/datasets/unified_holdout_v1/dataset.yaml`
 - 对照范围：`75` 张图、`150` 个 GT 框
 - Test 指标：
  - Precision：`0.9797`
  - Recall：`0.9653`
  - mAP50：`0.9875`
  - mAP75：`0.8773`
  - mAP50-95：`0.8042`
 - 单帧复盘口径：`conf=0.45`、`NMS IoU=0.7`、`match IoU=0.5`
 - 单帧复盘结果：`TP=144 / FP=3 / FN=6`

 #### 2.4.2 new clothes 可用基线

 - 当前 run：`clothes_merged_with_new_labels_v1_baseline`
 - split：`train=2106 / val=450 / test=453`
 - Test 指标：
  - Precision：`0.9835`
  - Recall：`0.9683`
  - mAP50：`0.9924`
  - mAP75：`0.8491`
  - mAP50-95：`0.7075`

 说明：这条线和上面的 `unified_holdout_v1` 不是同一个测试口径，因此不能直接把分数横向等同理解。

 #### 2.4.3 person 相关关键线

 - `person_fullframe_baseline`（旧 502 图）
  - Test：Precision `0.9228` / Recall `0.6740` / mAP50 `0.7606` / mAP50-95 `0.4102`
- `person_fullframe_with_new_labels_baseline`
  - Test：Precision `0.9304` / Recall `0.8552` / mAP50 `0.9054` / mAP50-95 `0.4802`
- `person_fullframe_with_new_labels_img768`
  - Test：Precision `0.8798` / Recall `0.8446` / mAP50 `0.8948` / mAP50-95 `0.4970`
  - 结论：`img768` 的 `mAP50-95` 更高，但 Precision / Recall / mAP50 不如 `640` 稳健基线，因此当前更适合作为“候选”，不是已替代主线。
- `person_roi_aware_v3_mask_then_crop_margin64_from_fullframe`
  - Test：Precision `0.9208` / Recall `0.7075` / mAP50 `0.7779` / mAP50-95 `0.4607`

 ### 2.5 当前总体判断

 从另一个项目组复用角度看，当前仓库最有参考价值的不是某一次单独高分，而是已经形成了下面这套闭环：

 - 数据路径与配置入口分离；
 - 数据集 prepare 与训练入口分离；
 - clothes 与 person 各自有完整主线；
 - ROI-aware person 有单独 prepared 数据构建链；
 - 训练后不是只看 mAP，还会做逐图 FP/FN 复盘与人工复核导出。

 ## 3. 各个 Python 文件的作用

 这一部分按模块来讲，便于后续人员知道“应该先看哪个文件”和“这个文件到底是主链还是辅助脚本”。

 ### 3.1 根目录通用脚本

 #### `backend-train-model/config.py`

 - **类型**：配置底座
 - **主要作用**：统一管理 clothes 默认数据路径、类别、切分比例、默认训练参数、本地权重路径和 Python 解释器候选。
 - **关键输入**：`project_config.json`
 - **关键输出**：运行时配置快照与默认参数集合
 - **在流程中的位置**：最前置的配置层
 - **应该怎么理解**：如果后续要改默认训练参数、默认数据入口、默认类别名，优先先看这里和 `project_config.json` 的映射关系。

 #### `backend-train-model/dataset_tools.py`

 - **类型**：数据准备主模块
 - **主要作用**：完成数据审计、标签校验、切分、prepared 数据集落盘。
 - **核心能力**：
  - 校验图片/标签一一对应；
  - 检查 YOLO 标注字段合法性；
  - 按 `sequence_contiguous` 或 `sequence_holdout` 生成 split；
  - 输出标准 YOLO 目录和 `dataset.yaml`。
 - **关键输入**：原始图片目录、标注目录、类别表、切分策略
 - **关键输出**：`prepare_report.json`、prepared 数据集目录、`dataset.yaml`
 - **在流程中的位置**：根目录 clothes 训练的核心数据准备层
 - **备注**：这是整个通用 clothes 链里最关键的底层模块之一。

 #### `backend-train-model/generate_split_manifests.py`

 - **类型**：切分 manifest 生成工具
 - **主要作用**：根据 source manifest 生成更平衡的 train/val 与统一 holdout split CSV。
 - **关键输入**：canonical/source manifest、分层规则、随机种子
 - **关键输出**：`trainval_balanced_v1.split.csv`、`unified_holdout_v1.split.csv`
 - **在流程中的位置**：merged clothes 训练前的 split 设计层
 - **适用场景**：当数据已经不是简单单源 sequence，而是多 source、多角色样本时，就不能只靠默认 contiguous 切分，需要用这类显式 manifest 工具。

 #### `backend-train-model/build_merged_clothes_dataset.py`

 - **类型**：多源 clothes 数据构建脚本
 - **主要作用**：把多组 clothes 数据源按 build 配置合并成统一 YOLO 数据集。
 - **核心能力**：
  - 读取 `*.build.json`；
  - 按 `split_manifest_csv` 强约束样本归属；
  - 处理缺失标签策略；
  - 输出合并后的 `dataset.yaml`、`manifest.csv`、构建摘要。
 - **关键输入**：`All-train-model/*.build.json` 或 `new_clothes_train/*.build.json`
 - **关键输出**：标准化 merged 数据集目录
 - **在流程中的位置**：merged clothes 主线的数据准备核心层

 #### `backend-train-model/train_workwear.py`

 - **类型**：clothes 训练总入口
 - **主要作用**：把 `audit -> prepare -> train -> evaluate -> export` 串成一条可直接运行的 clothes 训练流程。
 - **命令层特点**：支持 `audit / prepare / train / evaluate / export / all`
 - **关键输入**：`project_config.json`、prepared `dataset.yaml`、本地模型权重
 - **关键输出**：训练 run、评估报告、导出权重、总报告
 - **在流程中的位置**：通用 clothes 链的总调度器
 - **备注**：如果要快速看懂 root 目录 clothes 是怎么训练起来的，优先看这个文件。

 ### 3.2 `All-train-model/` 下的 Python 脚本

 #### `backend-train-model/All-train-model/analyze_baseline_fpfn.py`

 - **类型**：评估复盘脚本
 - **主要作用**：对当前 clothes baseline 的 val/test 结果做逐图 FP/FN 分析。
 - **关键输入**：baseline 评估报告、`best.pt`、`dataset.yaml`
 - **关键输出**：`baseline_fpfn_per_image.json`、`baseline_fpfn_summary.md`
 - **在流程中的位置**：训练完成后的复盘层
 - **备注**：这是结果解释脚本，不是训练入口脚本。

 ### 3.3 `new_clothes_train/` 下的 Python 脚本

 #### `backend-train-model/new_clothes_train/train-code/prepare_new_clothes_dataset.py`

 - **类型**：新增工服样本接入脚本
 - **主要作用**：把新增 clothes 样本并入现有 merged 体系，并尽量保持旧 7 源的 split 口径不被打乱。
 - **核心逻辑**：
  - old 7 sources 继续沿用 legacy manifest；
  - 新源 `gnew` 单独补齐缺失标签；
  - 新源按 `positive / empty` 分层随机切分。
 - **关键输入**：旧 manifest、新样本图片、原始标签目录
 - **关键输出**：完成标签目录、新 split CSV、summary JSON
 - **在流程中的位置**：new clothes 扩样主链的数据准备层
 - **备注**：专项脚本，路径与这轮新增数据绑定较强。

 #### `backend-train-model/new_clothes_train/train-code/validate_new_clothes_source.py`

 - **类型**：源数据质检脚本
 - **主要作用**：检查新增工服样本的图片可读性、YOLO 标签格式、空标注情况和孤立标签问题。
 - **关键输入**：新增 source 的图片与标签目录
 - **关键输出**：JSON 和 Markdown 版校验报告
 - **在流程中的位置**：new clothes 接入前后的质量检查层
 - **备注**：辅助性很强，但在接新数据时非常实用。

 ### 3.4 `person-train-model/` 下的 Python 脚本

 这一组脚本最完整，已经形成了“数据准备 -> ROI 衔接 -> 训练 -> 逐图复盘 -> 人工复核导出”的闭环。

 #### `backend-train-model/person-train-model/train-code/prepare_person_dataset.py`

 - **类型**：person 配置与标签汇总底座
 - **主要作用**：解析 `person_project_config*.json`，汇总多序列 person 标签，并给缺失标签图片补空白负样本。
 - **关键输入**：`person_project_config*.json`
 - **关键输出**：聚合标签目录、source summary JSON、运行上下文对象
 - **在流程中的位置**：person 全链路最前置的数据准备层

 #### `backend-train-model/person-train-model/train-code/prepare_roi_aware_person_dataset.py`

 - **类型**：ROI-aware prepared 数据生成脚本
 - **主要作用**：把 fullframe person 数据按 ROI polygon 转成 ROI-aware person 数据集。
 - **核心逻辑**：
  - 读取 `roi_config.generated.json`；
  - 判断人框是否满足 keep rule；
  - 根据 `mask_then_crop` 或 `crop_only` 生成 prepared 图像；
  - 重新输出对应标签。
 - **关键输入**：person project config、ROI config、原始聚合标签
 - **关键输出**：ROI-aware prepared 数据集目录、`dataset.yaml`、prepare 报告
 - **在流程中的位置**：ROI-aware person 主链的数据准备核心层

 #### `backend-train-model/person-train-model/train-code/run_person_flow.py`

 - **类型**：person 训练总入口
 - **主要作用**：为 person fullframe / ROI-aware 提供统一 CLI 包装，不直接复用 root 的 clothes 入口命令风格。
 - **命令层特点**：支持 `prepare-labels / setup-roi-workdir / extract-roi-config / audit / prepare / prepare-roi-aware / train / evaluate / export / all`
 - **关键输入**：`person_project_config*.json`、prepared `dataset.yaml`、权重
 - **关键输出**：训练 run、评估报告、导出权重、ROI 数据产物
 - **在流程中的位置**：person 分支的总调度器
 - **备注**：如果要快速理解 person 线，优先看这个文件。

 #### `backend-train-model/person-train-model/train-code/setup_roi_workdir.py`

 - **类型**：人工 ROI 标注准备脚本
 - **主要作用**：为每个 person sequence 抽取代表帧，搭建 ROI 标注工作区。
 - **关键输入**：person project config、抽帧数量
 - **关键输出**：`roi-work/` 工作目录、代表帧、说明文档
 - **在流程中的位置**：ROI 人工标注之前
 - **备注**：它不负责训练，但负责把 ROI 标注工作组织起来。

 #### `backend-train-model/person-train-model/train-code/labelme_roi_to_config.py`

 - **类型**：ROI 标注转配置脚本
 - **主要作用**：把 Labelme 的 ROI JSON 转成后续 prepared 数据生成可直接使用的统一 ROI 配置 JSON。
 - **关键输入**：ROI JSON 根目录、person project config
 - **关键输出**：`roi_config.generated.json`
 - **在流程中的位置**：ROI 标注与训练链之间的桥接层

 #### `backend-train-model/person-train-model/train-code/visualize_roi_filter_samples.py`

 - **类型**：ROI 过滤逻辑可视化脚本
 - **主要作用**：把 ROI polygon、人框保留/丢弃原因、crop 边界等叠加到图上，便于肉眼复盘。
 - **关键输入**：project config、ROI config、指定问题帧 stem
 - **关键输出**：overlay 图像
 - **在流程中的位置**：ROI-aware 规则诊断阶段
 - **备注**：明显偏调试和复盘用途。

 #### `backend-train-model/person-train-model/train-code/analyze_person_fpfn.py`

 - **类型**：person 逐图 FP/FN 分析脚本
 - **主要作用**：对 person 模型在指定 split 上做逐图误检漏检分析。
 - **关键输入**：评估报告或 `dataset.yaml + best.pt`
 - **关键输出**：`fpfn_per_image.json`、摘要报告
 - **在流程中的位置**：训练后的评估复盘核心层
 - **备注**：这是 person 分支里非常关键的复盘入口脚本。

 #### `backend-train-model/person-train-model/train-code/review_roi_cropped_keep_boxes.py`

 - **类型**：专项 ROI 裁边诊断脚本
 - **主要作用**：专门找出“规则上应该保留、但被当前 crop 裁到”的人框，并模拟 margin 扩边的修复效果。
 - **关键输入**：project config、ROI config、margin 参数
 - **关键输出**：裁边复盘图与统计摘要
 - **在流程中的位置**：ROI 裁边专项复盘
 - **备注**：更偏阶段性问题分析，不是日常主流程。

 #### `backend-train-model/person-train-model/train-code/review_person_roi_failures.py`

 - **类型**：ROI-aware failure 复盘脚本
 - **主要作用**：把原图 ROI 信息、prepared 图、GT/Pred 结果整合成统一复盘画板。
 - **关键输入**：`fpfn_per_image.json`、project config、ROI config
 - **关键输出**：失败样本拼图和说明
 - **在流程中的位置**：hard FN 深度诊断阶段
 - **备注**：很适合用于人工复核讨论。

 #### `backend-train-model/person-train-model/train-code/export_hard_review_assets.py`

 - **类型**：人工复核资产导出脚本
 - **主要作用**：把多个 run 的复盘结果整理成按 source 分包的人工复核素材。
 - **关键输入**：prepare 报告、多个 run 的 `fpfn_per_image.json`
 - **关键输出**：by-source overlay、说明文件、打包素材
 - **在流程中的位置**：训练与复盘之后的人工评审资产整理层
 - **备注**：明显偏阶段性导出工具。

 #### `backend-train-model/person-train-model/train-code/annotate_by_source_overlay_legend_zh.py`

 - **类型**：可视化后处理脚本
 - **主要作用**：给已经生成的 overlay 图批量加中文图例。
 - **关键输入**：已有 overlay 图片目录
 - **关键输出**：带中文图例的 overlay 图片
 - **在流程中的位置**：人工复核材料美化和补充说明阶段
 - **备注**：辅助脚本，不参与训练逻辑。

 ## 4. 数据集切分方式，以及更有效的改进方向

 这一部分是最值得给另一个项目组参考的，因为当前仓库最大的经验之一就是：**真正影响后续可解释性和跨场景泛化的，往往不是先换更大模型，而是先把切分和评估口径设计好。**

 ### 4.1 当前各条训练线的切分方式总表

 | 训练线 | 主要配置入口 | 当前有效切分方式 | 默认比例 | 是否主要用 `sequence_contiguous` | 是否有显式 split manifest | 是否有显式 holdout |
 | --- | --- | --- | --- | --- | --- | --- |
 | 单源 clothes | `backend-train-model/project_config.json` | 3 个 sequence 各自按连续区间切 `train/val/test` | `0.70 / 0.15 / 0.15` | 是 | 否 | 否 |

 ### 4.2 为什么当前很多地方仍用 `sequence_contiguous`

 当前项目早期之所以大量使用 `sequence_contiguous`，有它的现实原因：

 1. 早期 sequence 数量不多；
 2. 每条 sequence 内样本量相对较大；
 3. 如果直接按完整 sequence 做 holdout，训练集会缩得比较厉害；
 4. 与完全随机打散帧相比，`sequence_contiguous` 至少保留了“时间连续片段”的基本结构。

 它的优点是：

 - 每条 sequence 都能给训练集贡献样本；
 - 比全局随机帧切分更尊重视频时间结构；
 - 代码实现简单，复现实验也方便。

 但它的局限同样明确：

 - `train/val/test` 仍然来自同一条 sequence 的不同时间段；
 - 同一摄像头、同一背景、同一光照条件可能同时出现在不同 split；
 - 更像“同域泛化评估”，不是严格的跨场景泛化评估。

 对另一个项目组来说，这一点最值得注意：**如果是要迁移到加氢站新站点，不能只满足于这种切分。**

 ### 4.3 为什么 clothes merged 主线升级成了 manifest + unified holdout

 当前 clothes 主线之所以更成熟，就是因为它已经从“默认比例切分”升级成了“显式 sample-level split manifest”。

 这样做的好处是：

 - 不同实验 run 使用的是同一套 train/val/test 口径；
 - baseline 对比更公平；
 - 回滚路径更清晰；
 - 后续 route verification 或真实链路验证时，测试集不会偷偷变化。

 当前 `unified_holdout_v1` 虽然规模只有 `75` 张图，但它已经具备一个重要价值：

 > 它是一个可被长期复用的统一对照集，而不是每次 prepare 数据都顺便重新切出来的 test 集。

 ### 4.4 为什么 new clothes 对 `gnew` 改成了分层随机

 当前仓库已经遇到一个很典型的问题：新样本 `gnew` 本身不是单一连续视频，而是多段视频或多场景拼接后形成的 flat source。

 如果继续把它当作一个真正的单 sequence 做 contiguous 切分，会带来明显的分布失衡风险。

 因此当前做法是：

 - legacy 旧 7 源不动；
 - 新源 `gnew` 单独按 `positive / empty` 分层随机切分；
 - 先把明显不合理的 contiguous 假设去掉。

 这比错误地把 flat source 当单 sequence 更稳，但它仍然只是“务实修补版”，不是最终最佳答案。

 ### 4.5 为什么 ROI-aware person 还没有上 manifest

 当前 ROI-aware person 的核心任务不是“做终局评测体系”，而是“在同一批旧 502 图上做尽量干净的 ROI 规则对照”。

 所以当前它更强调：

 - 数据集来源不变；
 - image-level split 不变；
 - 只改 ROI keep rule、ROI mode、crop margin。

 这样才方便解释 `v1 -> v2 -> v3` 的差异到底来自哪一项规则变化。

 这对研究和复盘很有价值，但它也意味着：

 - 当前 ROI-aware test 更偏 native test；
 - 它还不是严肃的跨场景泛化评估终态；
 - 对外迁移到新场站时，不能直接把当前分数当成跨域结论。

 ### 4.6 更有效的改进方式：按优先级给建议

 下面这些建议，不是泛泛而谈，而是按当前仓库现状排序后的更高收益方向。

 #### 第一优先级：为新场站建立显式样本 manifest

 建议至少给每张样本补齐如下元数据：

 - `site_id`
 - `camera_id`
 - `sequence_name`
 - `time_of_day` 或班次信息
 - `source_id`
 - 对 clothes 样本额外补 `style_tag`

 这样做的收益是：

 - 可以把 person 线也升级到 sample-level manifest；
 - 可以按摄像头、场站、时段做真正的 split；
 - 后续想做“新场站 holdout”“夜间 holdout”“远景 holdout”都更容易。

 #### 第二优先级：建立统一跨域 holdout

 如果是加氢站项目组接手，强烈建议不要只保留普通 test，而是单独建一份跨域 holdout，例如：

 - `hydrogen_holdout_v1`
 - 或 `cross_domain_holdout_v1`

 并且保留单位尽量不是单帧，而是：

 - 整摄像头；
 - 整时段；
 - 整场景片段；
 - 或整条 sequence。

 只有这样，后续回答“模型换到新站点还能不能打”时，结论才更可信。

 #### 第三优先级：clothes 继续单类，但补齐 `style_tag`

 对加氢站场景，不建议第一步就按工服款式拆类。

 更合理的做法是：

 - 类别仍然保持 `0 -> clothes`；
 - 额外记录 `style_tag / site_id / camera_id / source_id`；
 - 单独保留一部分新款或新站点样本做 holdout。

 这样可以先回答一个更关键的问题：

 > 在不拆类的情况下，模型能不能已经学会新款工服或新站点工服。

 #### 第四优先级：先做 seed 稳定性，再谈更大模型或更大输入尺寸

 当前 ROI-aware 主线自己已经给出很明确的下一步：

 - 先补 `seed=7 / seed=13`；
 - 再做 hardest FN 深化复盘；
 - 只有边界证据闭环了，才考虑 `min_box_ioa 0.25 -> 0.20`。

 这背后的经验很重要：

 - 小幅领先不一定稳；
 - 如果 seed 波动大于你看到的提升，那就不该太早下结论；
 - 继续放大 `imgsz` 或直接换 `yolov8s`，都不是第一优先级。

 #### 第五优先级：建立 person -> clothes 的级联离线评估

 当前系统真实上线时不是单模型，而是链路。因此离线评估不应该只看 clothes mAP 和 person mAP。

 对另一项目组最值得补的一环，是至少做下面三档对照：

 1. `clothes fullframe` 直接跑；
 2. `oracle person boxes -> clothes`；
 3. `pred person boxes -> clothes`，并区分：
  - `pred fullframe person`
  - `pred ROI-aware person`

 这样才能看清：

 - person 上游到底帮了 clothes，还是截断了 clothes 召回；
 - ROI-aware person 到底是对真实链路有利，还是只在 native test 上更好看。

 #### 第六优先级：把 hardest FN 回流机制标准化

 当前仓库已经有很好的基础资产：

 - clothes：`analyze_baseline_fpfn.py`
 - person：`analyze_person_fpfn.py`
 - ROI：`review_person_roi_failures.py`、`visualize_roi_filter_samples.py`

 建议加氢站项目组直接延续这套做法，把 hard case 按以下维度长期沉淀：

 - `crowded_overlap_cases`
 - `boundary_cases`
 - `far_small_cases`
 - `visibility_weak_cases`

 这样后续的样本回流、split 设计和规则调整都会更有依据。

 ## 5. 本项目后端加油站工服检测的具体思路，以及迁移到加氢站时的落地方案

 ### 5.1 当前项目的真实技术路线

 当前项目真实可落地的思路，不是“一个模型直接判断未穿工服”，而是分层路线：

 1. **clothes 检测子模型**：负责给出“是否检测到工服区域”的证据；
 2. **person fullframe 模型**：负责整图找人；
 3. **ROI-aware person 模型**：负责把“人”进一步聚焦到 ROI 内候选作业人员；
 4. **在线链路规则**：再把 `person + ROI + clothes + tracker + temporal rule` 串起来，形成实际的违规告警逻辑。

 用一句话概括就是：

 > 当前更准确的系统表述，是“基于 ROI 内候选人员与工服证据的时序合规判断链路”，而不是“已经完成版的工人身份识别 + 未穿工服终局系统”。

 ### 5.2 后端训练侧三条核心线

 #### 5.2.1 clothes 检测子模型

 - 当前任务语义：单类 `clothes`
 - 当前主优势：
  - 数据治理比较成熟；
  - 已有 merged manifest；
  - 已有统一 holdout；
  - baseline 已固定，回滚路径清晰。
 - 当前能力边界：
  - 它只是在图像上给出工服检测框；
  - 它本身不判断这个人是不是工人，也不判断是不是该报警。

 #### 5.2.2 person fullframe

 - 任务语义：整图单类 `person`
 - 当前现实分支：
  - 旧 502 图 baseline；
  - 加入 `new_person_labels` 后的扩样 fullframe 线。
 - 当前价值：
  - 是 ROI-aware 的上游初始化来源；
  - 也是在 ROI 未补齐前最容易先扩样的主线。

 #### 5.2.3 ROI-aware person

 - 任务语义：在 ROI 业务约束下更聚焦作业区人员
 - 当前核心做法：
  - 先做人框检测；
  - 用 ROI polygon 决定哪些人框保留；
  - 再用 `mask_then_crop` 或 `crop_only` 生成更聚焦的 prepared 数据；
  - 然后重新训练 person 模型。
 - 当前最推荐版本：
  - `ROI-aware v3`：`mask_then_crop + crop_margin_px=64`
 - 当前结论：
  - 相比历史 ROI-aware baseline 明显更好；
  - 相比 `v2` 只有小幅优势；
  - 当前主瓶颈已经不再像是继续加 crop margin。

 ### 5.3 与在线告警链路的衔接方式

 虽然这份文档主要讲 `backend-train-model`，但另一个项目组如果要真正落地，必须知道训练产物最终怎么接到线上。

 当前在线链路参考口径是：

 1. person 模型先找人；
 2. ROI 规则筛出 ROI 内候选作业人员；
 3. clothes/workwear 模型判断该目标附近是否有工服证据；
 4. tracker 对同一人做短时跟踪；
 5. temporal rule 再决定是否报警。

 当前线上参考阈值包括：

 - `PERSON_CONF = 0.55`
 - `WORKWEAR_CONF = 0.45`
 - `ROI_MIN_OVERLAP_RATIO = 0.5`
 - `TEMPORAL_WINDOW_SIZE = 5`
 - `TEMPORAL_TRIGGER_RATIO = 0.6`
 - `MIN_TRACK_APPEAR_FRAMES = 2`
 - `TRACKER_MAX_AGE = 2`

 这里最重要的不是数字本身，而是它体现的系统思想：

 - 不把单帧结果直接当最终告警；
 - 不把工服检测直接当工人身份识别；
 - 用 ROI、轨迹和时序规则去做业务层收口。

 ### 5.4 如果迁移到加氢站，推荐的落地实施步骤

 对加氢站项目组来说，建议按下面顺序落地，而不是一上来就换更大模型。

 #### 阶段 1：先统一数据组织方式

 建议先把数据组织成下面这种结构性信息：

 - 场站：`site_id`
 - 摄像头：`camera_id`
 - 时间片段：`sequence_name`
 - 班次 / 时段：`shift` 或 `time_of_day`
 - clothes 样本补 `style_tag`

 图片和标签仍保持 YOLO 检测格式：

 - clothes：`0 -> clothes`
 - person：`0 -> person`

 先不要因为工服款式变化就拆类。

 #### 阶段 2：先复用 clothes 单类思路做 mixed baseline

 具体建议：

 - 新旧工服都先统一标成 `clothes`；
 - 用类似 `build_merged_clothes_dataset.py` 的思路生成 merged 数据；
 - 为 train/val/test 建立显式 manifest；
 - 单独留一部分新站点/新款工服样本做 holdout。

 目标是先回答：

 > 在不拆类的情况下，模型能不能已经学会新场站工服外观。

 #### 阶段 3：训练 person fullframe

 先训练 fullframe person，而不是直接把所有精力压在 ROI-aware。

 原因是：

 - fullframe person 更容易先扩样；
 - ROI 标注不全时不容易误接入；
 - 后续无论是否上 ROI-aware，fullframe person 都是重要上游能力。

 #### 阶段 4：如果 ROI 可定义，再训练 ROI-aware person

 只有在加氢站场景里 ROI 业务边界比较明确时，才建议继续复用 ROI-aware 路线。

 可复用的经验包括：

 - 先做 ROI workdir；
 - 再把 Labelme ROI 转统一配置；
 - 先做 `mask_then_crop` 主线；
 - 先沿用 `bottom_center_inside OR box_ioa >= 0.25`；
 - 优先补 seed 与 hard FN 复盘，而不是立刻改 keep rule。

 #### 阶段 5：离线评估不要只看单模型 mAP

 建议至少保留四类评估：

 1. clothes 单模型 native eval；
 2. person 单模型 native eval；
 3. ROI-aware person native eval；
 4. `person -> clothes` 级联离线评估。

 对实际报警有价值的，往往是第 4 类。

 #### 阶段 6：上线前补规则层验证

 最终上线时，需要在 `inspection-flask` 类似链路里做：

 - ROI 过滤；
 - track 关联；
 - temporal 去抖；
 - 阈值标定；
 - 报警样例回看。

 这一步说明：

 - `backend-train-model` 交付的是“模型和训练资产”；
 - 业务上线交付的是“链路”，两者不能混为一谈。

 ### 5.5 当前能力边界，必须对外写清楚

 如果这份文档要发给别的项目组，建议把下面几条边界直接保留：

 1. **当前 clothes 模型不是完整未穿工服告警器**；
 2. **当前 person + ROI 不是工人身份识别系统**；
 3. **当前 ROI-aware v3 相对 v2 是小优势，不是压倒性领先**；
 4. **当前 new labels person 主线仍是 fullframe，尚未完成 new labels ROI-aware 正式闭环**；
 5. **当前大多数分数更偏加油站域内结论，不能直接视作对加氢站已完成跨域验证。**

 ## 7. 给加氢站项目组的直接复用建议

 ### 7.1 可以直接复用的部分

 - YOLO 检测格式与数据配对规则；
 - clothes 多源 merged 数据构建思路；
 - 显式 split manifest 与 unified holdout 设计；
 - person fullframe 与 ROI-aware person 的双线结构；
 - 逐图 FP/FN 复盘脚本思路；
 - 人工复核导出与 hard case 回流方式。

 ### 7.2 需要按加氢站场景改造的部分

 - 数据路径、场站编号、摄像头编号与 ROI 定义；
 - split 设计，尤其是跨域 holdout；
 - 工服新款式的样本元数据，而不是先拆多类；
 - 在线阈值，不建议直接照抄当前加油站阈值；
 - 是否需要更严格的夜间、逆光、远景、小目标专门评估。

 ### 7.3 不建议原样照搬的部分

 - 把 flat source 直接当一条单 sequence；
 - 只看单模型 mAP 就判断能否上线；
 - 未做 seed 稳定性就认定小幅提升成立；
 - 把 `clothes` 检测子模型直接写成“完整未穿工服告警模型”；
 - 把 `person + ROI` 写成“已完成工人身份识别”。

 ---

 ## 8. 一句话总结

 当前 `backend-train-model/` 最值得另一个项目组学习的，不是某一个 run 的分数，而是它已经形成了下面这套相对完整的后端训练方法论：

 - 用配置文件而不是硬编码组织训练入口；
 - 用 prepared 数据和 manifest 管理切分口径；
 - 把 clothes 和 person 拆成可独立演进的两条子线；
 - 在 ROI-aware person 上把“规则设计、prepared 数据、误差复盘”连成闭环；
 - 在真正上线前，用 `person -> ROI -> clothes -> track -> temporal` 的系统视角看问题，而不是把单模型分数直接当最终业务能力。

 如果迁移到加氢站，最应该先复用的是这套方法论；最不应该先做的，是一上来就把问题简单理解成“换个更大的模型”。
