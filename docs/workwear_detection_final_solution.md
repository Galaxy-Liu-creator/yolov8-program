# 加油站工人工服检测最终方案说明

## 文档定位

本文用于沉淀当前讨论得到的**最终推荐解决方案**，并详细说明整条链路中的每一步、每一步的输入输出、为什么需要这一层，以及它与当前代码的关系。

本文中的“最终方案”指的是：

- **面向真实生产的推荐落地方案**
- 不是简单复述当前代码
- 也不是假设已经具备完整工人身份识别能力后的理想终局

本文推荐的当前生产落地链路为：

`person -> track_id -> ROI / area / stay filtering -> candidate worker -> workwear detection -> track-level temporal verification -> alarm`

长期升级目标为：

`person -> worker identity -> workwear detection -> temporal verification -> alarm`

---

## 一、为什么要采用这条链路

### 1.1 真实生产目标是什么

项目最终面向的真实业务目标是：

**加油站工人未穿戴工服检测**

这意味着系统最终想判断的是：

1. 这个目标是不是工人
2. 如果是工人，他有没有穿戴工服

### 1.2 为什么当前不能直接写成“先识别工人，再检测工服”

因为当前代码和数据层面还没有真正成熟的：

- worker/customer 独立身份分类模型
- 工牌 / 门禁 / 工位 / 排班等辅助身份信号
- 稳定可用的员工身份确认模块

当前项目现有显式能力主要是：

- `person`
- `ROI`
- `track_id`
- `clothes / workwear`
- `temporal rule`

因此，如果现在硬把系统主链路写成“已先识别工人”，就会出现两个问题：

- 业务表述超过当前真实能力边界
- 一旦前面的“疑似工人判断”不稳定，后续工服检测会直接漏掉目标

### 1.3 为什么也不能只停留在“ROI 内人员工服检测”

因为真实生产要解决的，不是单纯“ROI 内任何人没穿工服”；
而是更贴近：

- 在加油站作业区里
- 与工作行为高度相关的目标
- 是否未按要求穿戴工服

所以最合理的当前解法，不是两个极端中的任意一个，而是中间的生产过渡方案：

- 先做稳定的人体跟踪
- 再根据 ROI、面积、停留、轨迹稳定性形成“候选作业人员”
- 再对候选目标做工服检测和时序告警

---

## 二、方案总览

### 2.1 当前推荐生产链路

```text
视频帧输入
  -> person 检测
  -> track_id 分配
  -> ROI / area / stay filtering
  -> candidate worker
  -> workwear detection
  -> track-level temporal verification
  -> alarm / evidence / suppression
```

### 2.2 长期升级链路

```text
视频帧输入
  -> person 检测
  -> worker identity
  -> workwear detection
  -> temporal verification
  -> alarm
```

当前方案中的 `candidate worker` 是一个**可升级接口**：

- 当前阶段由规则产生
- 后续可由 worker/customer 模型输出替换

这样可以保证：

- 当前可以落地
- 后续不需要推翻整条链路

---

## 三、链路逐步说明

## 3.1 步骤 0：视频帧采集

### 作用

- 从摄像头流中持续取最新帧
- 为后续检测链路提供输入

### 输入

- RTSP / 海康摄像头实时视频流

### 输出

- 单帧图像 `frame`
- 对应时间戳 `timestamp`

### 为什么需要这一层

- 所有后续判断都必须有统一的帧和时间基线
- `track_id`、停留时长、时间窗告警都依赖准确时间信息

### 当前代码关系

当前线程会从缓存中读取当前摄像头最新帧，并记录时间戳。

---

## 3.2 步骤 1：person 检测

### 作用

- 从整帧中先找出所有人

### 输入

- 单帧原图

### 输出

- 每个人的：
  - `bbox`
  - `confidence`
  - `label`

### 为什么必须先做人

- 工服检测不应该对整帧盲扫
- 真实业务关心的对象是“人”
- 只有先找到人，后面才有可能围绕“人”建立工服合规判断

### 当前代码关系

- 在线链路已具备 `person` 检测
- 离线验证链路也已复用该能力

---

## 3.3 步骤 2：`track_id` 分配

### 作用

- 把跨帧看起来是同一个人的目标绑定成同一个 track

### 输入

- 当前帧的人体框列表
- 上一轮历史 track 状态

### 输出

- 带 `track_id` 的人目标列表

### 为什么这一层非常关键

真实摄像头场景中，单帧结果永远是不稳定的：

- 遮挡
- 转身
- 抖动
- 光照变化
- 偶发漏检

如果没有 `track_id`，就只能按单帧做逻辑，误报和漏报都会很高。

有了 `track_id` 之后，系统才能回答：

- 这是不是同一个人
- 他在 ROI 内待了多久
- 他连续多少帧没有检出工服

### 当前代码关系

当前代码已经有 IoU 跟踪器和 `track_id` 机制，这是现有实现里最值得保留的核心基础能力之一。

---

## 3.4 步骤 3：基础有效性过滤（面积 / 质量）

### 作用

- 去掉太小、太远、信息不足的人体框

### 输入

- 带 `track_id` 的 person 列表

### 输出

- 基础有效 person 列表

### 为什么需要这一层

过小的人体框通常存在这些问题：

- 工服信息根本看不清
- ROI 判断不稳定
- 很容易造成误检和无效工服推理

### 建议字段

- `area`
- `is_valid_person`
- `invalid_reason`

### 当前代码关系

当前代码已经有面积过滤，但建议把它和后续业务判断拆得更清楚。

---

## 3.5 步骤 4：ROI 过滤

### 作用

- 判断目标是否处于受监管的作业区域内

### 输入

- 基础有效 person
- 摄像头 ROI 配置

### 输出

- `in_roi`
- `roi_overlap_ratio`

### 为什么需要这一层

真实生产不会对整幅图里的所有人做同等监管。

ROI 的作用是：

- 把监管范围收束到真正的作业区
- 降低顾客和无关行人的干扰
- 降低无效工服检测的算力开销

### 关键注意点

- ROI 不能画得太大
- 不能把顾客常驻区、车道等待区、便利店入口等大面积包进去
- ROI 本身不是员工身份识别，只是一个业务约束

---

## 3.6 步骤 5：停留 / 稳定出现判断

### 作用

- 判断目标是否只是短暂掠过，还是形成了稳定监管对象

### 输入

- `track_id`
- `in_roi`
- 连续帧出现信息

### 输出

- `stay_frames`
- `stay_seconds`
- `is_stable_target`

### 为什么需要这一层

真实视频里经常会出现：

- 顾客从 ROI 边缘一晃而过
- 检测框偶然抖进 ROI
- 某个目标只出现 1 帧或 2 帧

这类目标不应立刻进入最终工服告警逻辑。

### 这一层解决的问题

- 过滤短暂误入
- 过滤边界抖动
- 为后续“候选作业人员”提供更稳定依据

---

## 3.7 步骤 6：候选作业人员判断（`candidate worker`）

### 作用

- 在当前没有真实 worker/customer 模型的前提下，用规则形成“需要接受工服检查的人”

### 输入

- `track_id`
- `in_roi`
- `roi_overlap_ratio`
- `stay_frames / stay_seconds`
- 面积 / 有效性

### 输出

- `candidate_worker = True / False`
- `candidate_reason`

### 为什么要引入这一层

这是整个当前方案最关键的“中间妥协层”：

- 它承认系统还没有真实员工识别能力
- 但又比“ROI 内所有人都直接做工服检测”更贴近生产目标

也就是说，它不是“已识别工人”，而是：

- 高概率与作业相关
- 应进入工服检查

### 为什么不能直接叫“员工识别”

因为它依赖的仍然是：

- 规则
- ROI
- 停留
- 轨迹稳定性

而不是独立身份模型。

所以当前更准确的命名应是：

- 候选作业人员
- 需工服检查的目标

---

## 3.8 步骤 7：工服检测

### 作用

- 对进入候选阶段的目标做工服检测

### 输入

- 候选作业人员的人框裁剪图

### 输出

- `workwear_items`
- `has_workwear`
- `workwear_checked`

### 为什么工服检测要放在这里

原因有三个：

1. 先做人框裁剪，能让工服检测更聚焦
2. 先筛掉明显无关目标，能减少无意义推理
3. 对真实摄像头部署更省算力

### 当前与未来的差别

- 当前阶段：只对候选作业人员做工服检测
- 未来阶段：对“已确认工人”做工服检测

因此这一步在架构上可以保持不变，只替换它的前置判断来源。

---

## 3.9 步骤 8：工服缺失的 track 级时序确认

### 作用

- 判断某个 track 是否在一个时间窗内持续表现为“未检出工服”

### 输入

- `track_id`
- `has_workwear`
- 时间窗历史记录

### 输出

- 某个 track 是否达到最终告警条件

### 为什么必须保留这一步

因为真实生产里，工服检测在单帧层面很容易波动：

- 转身时只看到背面
- 被油枪、车辆、顾客部分遮挡
- 某一帧清晰，下一帧模糊

如果没有时序确认，就会变成：

- 一帧没检出工服就报警

这在生产里几乎一定会造成大量误报。

### 推荐逻辑

- 只统计候选作业人员的 track
- 在时间窗内统计：
  - `appear`
  - `violation`
- 当 `violation / appear >= threshold` 时再触发

### 当前代码关系

当前 `vio_workwear_missing.py` 已经有按 `track_id` 聚合统计的规则，这部分非常适合保留并扩展。

---

## 3.10 步骤 9：告警、证据图与抑制窗口

### 作用

- 在真正满足条件时输出可运营、可复核的最终结果

### 输入

- 触发告警的 `track_id`
- 证据帧
- 规则名称
- 摄像头信息

### 输出

- 告警事件
- 证据图
- 数据库记录 / 内存事件

### 为什么需要抑制窗口

真实生产里，如果不做抑制，一个目标在 10 秒内可能连续报很多次。

抑制窗口的作用是：

- 避免重复报警
- 减少运营噪音
- 减少证据图冗余

### 当前阶段建议告警文案

- `作业区候选作业人员疑似未穿工服`
- `作业区目标疑似未按要求穿戴工服`

不建议当前直接写成：

- `已确认工人未穿工服`

除非系统已经具备可靠身份识别能力。

---

## 四、每一步的输入输出汇总

### 4.1 person 阶段

输入：

- `frame`

输出：

- `bbox`
- `confidence`
- `label`

### 4.2 track 阶段

输入：

- 当前帧 person 列表
- 历史 track

输出：

- `track_id`

### 4.3 ROI / area / stay 阶段

输入：

- 带 `track_id` 的 person
- ROI 配置
- 历史出现信息

输出：

- `in_roi`
- `roi_overlap_ratio`
- `area`
- `stay_frames`
- `stay_seconds`

### 4.4 candidate worker 阶段

输入：

- 上述全部基础字段

输出：

- `candidate_worker`
- `candidate_reason`

### 4.5 workwear 阶段

输入：

- 人框裁剪图

输出：

- `workwear_items`
- `has_workwear`

### 4.6 temporal 阶段

输入：

- 候选目标的 `track_id`
- `has_workwear`

输出：

- 是否达到最终告警阈值

### 4.7 alarm 阶段

输入：

- 触发 track
- 证据帧

输出：

- 事件
- 图片
- 日志 / 落库

## 五、为什么这条链路比两个极端方案更合适

### 5.1 为什么不直接用“ROI 内所有人都做工服检测”

缺点：

- 顾客误报容易高
- 算力浪费大
- 业务含义不够接近“工人未穿工服”

### 5.2 为什么不直接宣称“先识别工人，再检测工服”

缺点：

- 当前没有真实 worker/customer 能力
- 过早硬门控会放大漏报
- 文档表述会超过实现能力

### 5.3 为什么当前推荐链路更平衡

优点：

- 比单纯 ROI 规则更贴近真实生产目标
- 比强行身份识别更诚实、也更稳
- 与当前代码基础兼容
- 后续容易升级

## 七、现场部署时最应关注的问题

### 7.1 ROI 质量

- ROI 是否真的只覆盖作业区
- ROI 是否把顾客常驻区误包进来

### 7.2 候选阶段阈值

- 停留帧数是否过高导致漏报
- 停留帧数是否过低导致顾客误入

### 7.3 工服模型质量

- 背身、遮挡、低光环境下是否还能稳定识别

### 7.4 时序阈值

- 时间窗太短会误报
- 时间窗太长会漏掉短时违规

### 7.5 告警抑制

- 抑制过短会重复报
- 抑制过长会漏掉二次违规

---

## 八、后续升级路线

### 阶段 1

- 上线当前推荐过渡方案
- 积累真实误报 / 漏报样本

### 阶段 2

- 优化 candidate-worker 规则
- 引入更多作业行为和区域特征

### 阶段 3

- 训练 worker/customer 模型
- 将 `candidate worker` 替换为真实 `worker identity`

---

## 九、最终结论

当前最合理的最终推荐方案，不是两个极端中的任意一个：

- 不是“ROI 内任何人没检出工服就报”
- 也不是“现在就宣称已完成工人身份识别”

而是：

`person -> track_id -> ROI / area / stay filtering -> candidate worker -> workwear detection -> track-level temporal verification -> alarm`

这条链路的意义在于：

- 它更贴近真实生产目标
- 与当前代码基础兼容
- 可以先上线
- 可以持续演进
- 后续能平滑升级成真正的：

`person -> worker identity -> workwear detection -> temporal verification -> alarm`

因此，这条链路应作为当前版本的**正式推荐落地方案**。
