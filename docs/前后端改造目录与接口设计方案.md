# 前后端改造目录与接口设计方案

## 1. 文档目的

本文档用于给当前项目的前后端改造提供一版可直接落地的结构方案，重点覆盖三部分：

1. 目录结构设计
2. 数据库表设计方式
3. API 清单

本文档面向当前项目现状编写，默认遵循以下边界：

- 前端使用 Vue
- 后端提供 Web API、数据库访问和检测任务管理
- 算法部分从 `backend-python/main_test6.py` 拆分为 `detection_worker.py`
- 在线链路以 `person + ROI + clothes + track + temporal rule` 为主
- `backend-train-model/` 只作为训练产物和模型来源，不直接承担在线 Web 服务职责

## 2. 目标架构

建议将项目拆成四层：

1. 前端展示层
   Vue 页面，负责配置、展示和操作。
2. 后端接口层
   提供 REST API / WebSocket，负责参数校验、业务编排、权限控制。
3. 检测执行层
   使用 `detection_worker.py` 常驻运行模型，串行处理检测任务。
4. 数据与文件层
   数据库存储配置、事件和状态，文件系统存储截图、视频和日志。

这样拆分后，系统职责会更清楚：

- Vue 不直接接触模型
- API 不直接执行重型推理
- `detection_worker.py` 不负责页面逻辑
- 训练代码与在线服务彻底分开

## 4. 各目录职责建议

### 4.1 `backend-train-model/`

保持训练职责不变，只提供以下内容给在线系统使用：

- 已确认的模型权重路径
- 类别定义
- 当前推荐的模型版本
- 训练结果说明文档

不建议把训练脚本直接 import 到在线服务中执行。

### 4.2 `backend-service/app/api/`

这里只负责接口层：

- 接收前端请求
- 校验参数
- 调用 service
- 返回统一 JSON

不要在这里直接写模型推理逻辑。

### 4.3 `backend-service/app/services/`

这里放业务规则，例如：

- 摄像头配置保存
- ROI 配置保存
- 告警确认
- 告警记录查询
- 证据文件组织

### 4.4 `backend-service/app/inference/`

这里放纯算法运行时封装：

- 模型加载
- 推理
- 跟踪
- 时序规则判断

这里不直接处理 HTTP 请求，也不直接操作前端页面。

### 4.5 `backend-service/app/workers/`

`detection_worker.py` 是在线检测主执行类，建议作为常驻线程或后台任务运行。

它负责：

- 启动时加载模型
- 拉取摄像头配置
- 轮询各摄像头
- 串行执行检测
- 生成事件
- 调用 service 落库与存证

### 4.6 `frontend-vue/`

前端建议至少做以下页面：

- 登录页
- 首页概览页
- 摄像头管理页
- ROI 配置页
- 模型配置页
- 实时监控页
- 告警记录页
- 系统设置页

## 5. `detection_worker.py` 设计建议

### 5.1 定位

`detection_worker.py` 不应该是一个“接口函数”，而应该是一个“算法子线程类”。

建议类名示例：

```python
class DetectionWorker:
    ...
```

### 5.2 核心职责

建议它只做以下事情：

1. 初始化模型
2. 初始化视频流管理器
3. 按顺序轮询摄像头
4. 获取最新帧
5. 执行 `person -> ROI -> track -> clothes -> temporal`
6. 生成告警事件
7. 落库存证
8. 推送给前端

### 5.3 串行运行建议

既然当前要求“串行运行”，就建议采用“单 worker + 多摄像头轮询”的方式：

- 每轮只处理一个摄像头的一帧或一组抽样帧
- 不为每个摄像头各开一个模型线程
- 每路摄像头只保留最新帧，避免帧堆积

这样可以保证第一版结构稳定、调试简单。

### 5.4 不建议保留在 `main_test6.py` 中的内容

以下逻辑建议全部迁移出大脚本：

- 摄像头发现与取流
- 证据图视频保存
- 上传外部平台
- 音频播放
- 前端阈值同步
- 在线检测主循环

`main_test6.py` 可以保留为历史原型参考，但不应继续作为正式系统入口。

## 6. 数据库设计方式

## 6.1 设计原则

数据库设计建议遵循以下原则：

1. 配置与运行数据分离
   摄像头、ROI、模型、阈值属于配置；告警、心跳、日志属于运行数据。
2. 基础表与事件表分离
   基础表用于长期维护；事件表用于持续写入。
3. 文件只存路径，不直接存大二进制
   图片和视频保存在磁盘或对象存储，数据库保存路径与元信息。
4. 所有在线关键配置都应可追踪
   比如 ROI、阈值、启用状态、模型版本，最好带更新时间和版本信息。

## 6.2 数据库选型建议

正式项目建议：

- PostgreSQL

本地快速原型可用：

- SQLite

原因：

- SQLite 上手快，适合前期联调
- PostgreSQL 更适合后续多表查询、并发写入和长期维护

## 8. 表关系建议

推荐主关系如下：

- `camera` 1 对多 `camera_roi`
- `camera` 1 对多 `alarm_event`
- `alarm_event` 1 对多 `alarm_media`

建议原则：

- 配置表以 `camera` 为核心
- 事件表以 `alarm_event` 为核心
- 模型表和配置表作为全局辅助表

## 9. API 设计原则

建议 API 设计遵循以下规则：

1. 路径按资源划分，不按页面划分
2. 前后端统一返回格式
3. 查询和写入分离
4. 检测 worker 控制接口单独分组
5. 实时事件建议用 WebSocket 或 SSE 推送

统一返回格式示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

## 12. 第一阶段落地顺序建议

建议按以下顺序推进，而不是前后端同时无序开工。

### 第一步：先确定在线主链

先固定第一版只做：

- 摄像头取流
- person 检测
- ROI 过滤
- clothes 检测
- temporal rule
- 告警落库

### 第二步：先建后端骨架和数据库

先把这些搭起来：

- `backend-service/` 目录
- 数据库表
- 基础 API
- worker 状态控制

### 第三步：再拆 `main_test6.py`

把以下能力拆走：

- 摄像头流管理
- 检测循环
- 截图与录像
- 事件生成

目标是形成正式的 `detection_worker.py`。

### 第四步：最后接 Vue 页面

前端第一版先完成：

- 摄像头配置
- ROI 配置
- 模型切换
- 告警列表
- worker 状态展示

实时视频能力可以放到后面迭代。

## 13. 关键风险与建议

### 13.1 不要把 `clothes` 模型误写成“未穿工服模型”

当前在线逻辑必须明确：

- `clothes` 只表示检测到工服目标
- “未穿工服”是业务规则判断结果，不是单一模型输出

### 13.2 不要让 API 请求直接触发模型推理

正确方式是：

- API 只控制 worker
- worker 常驻运行模型

否则后续摄像头数量一多，系统会很快失控。

### 13.3 串行 worker 的上限要提前接受

串行方案优点是简单稳定，但限制也很明显：

- 摄像头越多，单路刷新间隔越长
- 必须做最新帧覆盖，不能做无限排队

因此第一版目标应放在“先稳定跑通”，不要过早追求大规模并发。

### 13.4 ROI 建议放在数据库动态配置

不要把 ROI 硬编码在脚本或静态配置文件里。

原因：

- 前端绘制和维护更方便
- 摄像头一多时更容易管理
- 后续可加版本控制和历史回滚

## 14. 建议结论

当前项目最合理的改造方向不是“继续扩写 `main_test6.py`”，而是：

1. 把训练目录与在线服务彻底分离
2. 新建正式后端服务目录
3. 用数据库承接摄像头、ROI、模型与运行参数
4. 用 `detection_worker.py` 承接串行算法运行
5. 用 Vue 页面承接配置、展示与告警管理

如果后续进入实现阶段，建议下一份文档继续细化：

- `detection_worker.py` 类设计
- 数据库字段最终版
- API 请求响应样例
- Vue 页面原型与字段对应关系
