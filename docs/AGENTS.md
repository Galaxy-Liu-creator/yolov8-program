# Docs 目录说明

本文件作用域覆盖 `docs/` 目录下的全部文档。

## 1. 文档定位

- `docs/` 用于记录数据集说明、代码分析、业务方案、汇报材料、论文材料和阶段总结。
- 文档帮助理解项目当前阶段，但不是不可质疑的固定真理。
- 当文档与真实代码、配置、训练结果、现场目标或用户最新说明冲突时，优先尊重真实情况。

## 2. 数据集文档

- 涉及数据集、标注、训练配置、转换、校验、可视化时，必须先读 `docs/dataset.md`。
- `docs/dataset.md` 是仓库级数据说明入口，至少应保持任务类型、类别表、标注格式、路径规则正确。
- 示例文件为 `docs/dataset_examples/sample_001.jpg` 与 `docs/dataset_examples/sample_001.txt`；如果缺失，依赖标注格式的代码修改前应提示用户补充。

## 3. 业务与方案写法

- 四类检测方向应作为同一项目的四个组成部分：工服检测、隔离栏缺失检测、打电话检测、吸烟检测。
- 不要在论文或汇报中写成“工服检测是主线，其他三个是拓展”。
- 可以通过篇幅和细节自然体现资料完整度差异，但文字上不要降低其他三类检测的项目地位。
- 当前在线工服链路应写成 `person + ROI + workwear + track-level temporal rule` 的候选作业人员工服合规判断，不要写成已完成真实工人身份识别。

## 4. 写作风格

- 学生汇报和论文材料优先使用通俗清楚的表述，少用堆叠式专业术语。
- 必要术语应保留，例如 `fullframe person`、`ROI-aware person`、`sequence_contiguous`、`unified holdout`、`mAP50-95`。
- 开发环境不需要写进论文正文；除非用户要求，不要说明文档是用 LaTeX 编写的。
- 更新业务 / 方案文档时，优先区分：真实生产目标、当前阶段实现、当前能力边界、升级触发条件、后续演进路线。

## 5. 维护规则

- 修改长期有效的业务口径、数据口径、训练状态、阶段结论时，同步检查根 `AGENTS.md` 是否需要更新。
- 如果某份文档引用了过时结论，应优先改成“历史阶段记录”或明确当前以哪个配置 / 报告为准。
- 不要把历史推测写成当前事实；指标、样本数、路径应尽量引用仓库中可核对的报告或配置。
