# ROI 与工服检测方案说明（生产版）

## 文档定位

本文用于记录：

- 真实生产目标
- 当前阶段可落地实现
- 与当前代码一致的执行链路
- 后续升级路线

本文**不是永久不变的最终架构结论**。  
当真实生产目标、现场约束、误报漏报情况、代码实现状态与现有文档不一致时，应优先以真实生产情况为准；Agent 可以提出质疑和修订建议，经用户批准后再同步更新文档。

## 一、真实生产目标

本项目面向真实摄像头接入与生产部署后的目标，应定义为：

**加油站工人未穿戴工服检测**

从业务语义上，理想链路应为：

`person -> 工人身份判断 -> 工服检测 -> 时序确认 -> 告警`

也就是说，最终希望系统判断的是：

- 这个目标是不是工人
- 如果是工人，是否未按要求穿戴工服

这一定义比“ROI 内人员是否检测到工服”更接近真实生产目标。

---

## 二、当前阶段实际能力边界

当前仓库在 `inspection-flask` 中已经具备的显式能力，主要是：

- `person` 检测
- ROI 约束
- `track_id` 级别的跨帧跟踪
- 工服 / `clothes` 检测
- 基于 track 的时序告警

当前**尚未具备**真正独立的：

- 工人 / 顾客身份分类模型
- 基于身份标签的稳定“员工确认”能力

因此，当前版本本质上仍然是一个**代理实现**：

`person + ROI + track_id + workwear + temporal`

它可以用于当前阶段的生产过渡落地，但**不等价于已经完成工人身份识别**。

---

## 三、当前代码的真实执行顺序

根据当前代码，在线链路更接近以下顺序：

`person -> 面积过滤 -> ROI 判断 -> 人框裁剪 -> 工服检测 -> 分配 track_id -> 按 track 做时序告警`

关键代码位置：

- `person` 检测：`inspection-flask/applications/common/hk_custom_threading_plus.py`
- IoU 跟踪器 / `track_id`：`inspection-flask/applications/common/hk_custom_threading_plus.py`
- ROI 判断：`inspection-flask/applications/common/hk_custom_threading_plus.py`
- 工服检测：`inspection-flask/applications/common/hk_custom_threading_plus.py`
- 按 `track_id` 做时序规则：`inspection-flask/violation_module/vio_workwear_missing.py`

这说明当前系统并不是“单帧 ROI 规则”，而是已经有明显的**track 级时序融合**能力。

---

## 四、当前阶段推荐上线链路

虽然当前代码里工服检测发生在 `track_id` 分配之前，但面向近期生产部署，推荐把当前阶段的正式上线链路定义为：

`person -> track_id -> ROI / 面积 / 停留过滤 -> 工服检测 -> 按 track 做时序告警`

这样定义的原因是：

- 更符合真实在线视频场景中的跨帧对象管理
- 能把 `track_id` 明确成核心中间状态
- 更方便后续插入“候选作业人员 / 候选员工”模块
- 不会误把当前版本说成已经具备真实工人身份识别能力

当前阶段建议把中间状态命名为：

- 候选作业人员
- 需工服检查的作业区人员

不建议直接命名为：

- 已确认工人
- 已识别员工

因为这会让系统表述超过当前真实能力边界。

## 五、最终目标链路 vs 当前过渡链路

### 5.1 最终目标链路

```text
person
  -> worker identity
  -> workwear detection
  -> temporal verification
  -> alarm
```

适用前提：

- 已有可靠的工人 / 顾客数据集
- 已具备独立 worker/customer 模型或等价身份信号
- 业务确实要求“必须只对工人报警”

### 5.2 当前过渡链路

```text
person
  -> track_id
  -> ROI / area / stay filtering
  -> workwear detection
  -> track-level temporal rule
  -> alarm
```

当前这条链路的业务含义应表达为：

- 当前版本先通过作业区规则筛出“需要检查工服的人”
- 再对这些目标做工服检测与时序确认
- 输出“作业区人员疑似未穿工服”告警

而不是：

- 已经准确识别出“这个人就是工人”

## 六、推荐的分阶段演进路线

### 阶段 1：当前可上线过渡版

目标：

- 先把真实摄像头场景下的告警链路跑稳
- 先积累误报、漏报和边缘样本

建议正式口径：

- 作业区人员工服合规检测
- 作业区人员疑似未穿工服

特点：

- 可以上线
- 与当前代码能力匹配
- 对遮挡、转身、抖动更稳
- 仍属于代理实现

### 阶段 2：候选作业人员增强版

目标：

- 在不具备真实 worker/customer 模型之前
- 用 ROI、停留、动作区域、出现频率、轨迹稳定性等规则
- 构建更稳的“候选作业人员”状态

推荐链路：

`person -> track_id -> candidate worker score -> workwear detection -> temporal verification -> alarm`

注意：

- 这一阶段仍然不应对外宣称“已完成工人识别”
- 它只是比单纯 ROI 更强的代理判断

### 阶段 3：真实工人身份判断版

目标：

- 引入 worker/customer 模型或其他身份辅助信号
- 真正把业务闭环推进到“工人未穿工服检测”

推荐链路：

`person -> worker identity -> workwear detection -> temporal verification -> alarm`

只有到这一阶段，系统才适合在技术表述上明确说：

- 先识别工人，再检测工服

---

## 七、什么时候必须从当前过渡方案升级

当出现以下情况时，应把“升级工人身份判断模块”列为高优先级事项：

- 顾客频繁进入 ROI
- 员工与顾客在 ROI 内高度混杂
- 现场误报主要来自“把顾客当成需检查对象”
- 业务要求系统必须明确区分员工与顾客
- 审计或处罚场景要求对“目标身份”给出更强可解释性

如果上述问题已经明显影响生产可用性，就不应再把当前代理方案当成长期终局。

## 八、建议的对外表述

### 8.1 当前阶段建议表述

- 作业区人员工服合规检测
- 作业区人员疑似未穿工服
- 作业区内目标未检测到工服

### 8.2 长期目标表述

- 加油站工人未穿戴工服检测
- 工人身份与工服合规联合检测

### 8.3 当前阶段不建议直接表述

- 已准确识别员工与顾客
- 已完成工人身份识别
- 进入 ROI 的人就是工人

## 十、当前阶段结论

当前版本最适合的结论不是：

- “系统最终只需要 ROI 内人员工服检测”

而应当是：

- **真实生产终局目标是“加油站工人未穿戴工服检测”**
- **当前阶段的可落地实现是“person + ROI + track_id + workwear + temporal”的代理方案**
- **后续应根据现场误报与样本积累情况，逐步升级到真实的工人身份判断模块**

换句话说：

- 真实生产目标，应支持“先判断是不是工人，再判断工服”
- 当前上线实现，应诚实表述为“先识别人，再通过作业区规则与时序做工服合规检测”
