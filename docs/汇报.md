# `inspection-flask` 项目阶段汇报

适用场景：`2026-04-03` 组内汇报  
整理时间：`2026-04-02`

## 1. 一句话结论

当前项目已经完成了 **工服检测主链 + 最小 Flask 后台骨架 + 离线诊断工具 + 软降级运行机制** 的主体搭建，已经从“没有主链”推进到“主链基本成形、正在补运行闭环和正确性闭环”的阶段。  
但**还不能直接对外宣称“系统已经完整可运行/可上线”**，当前最主要的缺口仍然是：

- Flask 服务依赖还未在默认环境补齐
- `inspection-flask/weights` 下的两套 YOLOv8 权重还未落地
- 真实海康原生 SDK 链路尚未完全恢复
- 数据库字段持久化与准确率验证还没有闭环

## 3. 项目目前已经完成了什么

### 3.1 业务目标已经收敛清楚

- 当前项目目标已经明确为：**加油站作业区人员工服检测**，不再继续恢复旧警务项目的全部业务能力。  
  证据：`docs/file_todo_list.md`、`docs/run_todo_list.md`
- 数据集口径已经明确：当前仓库文档描述的是 **单类 `clothes` 标注** 数据集。  
  证据：`docs/dataset.md`
- 离线评估默认参数已经同步收敛到当前数据集口径，`validate` 默认走 `clothes-only` 模式。  
  证据：`inspection-flask/main.py`、`docs/update_log.md`

### 3.2 YOLOv8 检测主链已经搭起来

- 已经完成 **人员检测模型 + 工服检测模型** 的双模型封装，并统一通过 `ultralytics.YOLO(...)` 加载。  
  证据：`inspection-flask/utils/models.py`、`inspection-flask/settings.py`
- 已经完成统一输出结构封装，检测结果统一整理成：
  - `bbox`
  - `confidence`
  - `label`  
  证据：`inspection-flask/utils/models.py`
- 已经完成工服合规策略统一封装，避免线上线下口径继续分叉。  
  证据：`inspection-flask/utils/workwear_policy.py`、`docs/Analysis_For_yolov8.md`

### 3.3 在线检测主线程已经具备完整的基本链路

- 在线检测链已经具备以下顺序：
  - 整帧人员检测
  - 人框裁剪
  - 工服二阶段检测
  - ROI 过滤
  - IoU 跟踪
  - 时间窗违规判定
  - 证据图保存  
  证据：`inspection-flask/applications/common/hk_custom_threading_plus.py`、`inspection-flask/violation_module/vio_workwear_missing.py`
- 当前 ROI 判定已经不是早期的“中心点命中”，而是**人框与 ROI 重叠比例**判定。  
  证据：`inspection-flask/applications/common/hk_custom_threading_plus.py`、`docs/Analysis_For_yolov8.md`
- 当前违规规则已经收敛为 **“作业区人员疑似未穿工服”**，避免过度宣称。  
  证据：`inspection-flask/settings.py`、`inspection-flask/violation_module/vio_workwear_missing.py`、`inspection-flask/applications/models/admin_violate_photo.py`

### 3.4 时序规则与告警存证已经建立

- 已经引入基于 `track_id` 的时序统计，不再简单按单帧直接报警。  
  证据：`inspection-flask/applications/common/hk_custom_threading_plus.py`、`inspection-flask/violation_module/vio_workwear_missing.py`
- 已经有：
  - `MIN_TRACK_APPEAR_FRAMES`
  - `TEMPORAL_TRIGGER_RATIO`
  - 告警抑制窗口  
  证据：`inspection-flask/settings.py`
- 证据图保存与违规写库链路已经保留，并适配当前工服场景。  
  证据：`inspection-flask/violation_module/base.py`、`inspection-flask/applications/view/system/hk_camera.py`

### 3.5 离线诊断与验证工具已经建立

- `main.py` 已支持：
  - `check`
  - `image`
  - `validate`  
  证据：`inspection-flask/main.py`

## 4. 项目目前还存在什么不足

### 4.1 还不能直接说“完整 Flask 服务已经能正常跑起来”

- `docs/check_log.md` 已明确给出结论：**当前机器上不能正常启动完整 Flask 服务**。  
  证据：`docs/check_log.md`
- 当前默认环境 `yolo_code` 已具备部分离线推理依赖，但缺少完整 Flask 服务依赖，例如：
  - `flask`
  - `sqlalchemy`
  - `flask_sqlalchemy`
  - `flask_marshmallow`
  - `marshmallow_sqlalchemy`
  - `apscheduler`
  - `psycopg2`  
  证据：`docs/check_log.md`、`docs/run_todo_list.md`
- 文档中已经记录过导入 `create_app()` 时首个阻断错误是：`ModuleNotFoundError: No module named 'apscheduler'`。  
  证据：`docs/check_log.md`

### 4.2 YOLOv8 权重文件还没有落地

- `settings.py` 已经声明了两套当前版本权重：
  - `inspection-flask/weights/person_detect_yolov8.pt`
  - `inspection-flask/weights/workwear_detect_yolov8.pt`  
  证据：`inspection-flask/settings.py`
- 但当前仓库里 **`inspection-flask/weights` 目录并不存在**。  
  证据：`docs/check_log.md`、`docs/run_todo_list.md`
- 这意味着：代码已经写好，但模型资源还没有真正接入当前项目目录。  

### 4.3 真实海康原生 SDK 链路还没有完全恢复

- 当前 `HKStream` 是“**最小兼容取流封装**”，优先解决的是本地图片 / 视频 / RTSP / OpenCV 可读流。  
  证据：`inspection-flask/hk/hksdk/device.py`
- 文档已经明确说明：**这不是“海康原生 SDK 已完整恢复”**。  
  证据：`docs/update_log.md`、`docs/file_todo_list.md`、`docs/run_todo_list.md`
- 也就是说，当前在线链路更偏向“先打通演示与验证”，离生产级海康 SDK 闭环还有距离。  

### 4.4 数据库字段持久化和数据闭环还未完成

- 当前已经支持通过 `camera_runtime_overrides` 在运行时注入：
  - `roi`
  - `frame_path`
  - `stream_url`  
  证据：`inspection-flask/applications/__init__.py`、`inspection-flask/applications/view/system/hk_camera.py`
- 但文档已明确说明：**当前采用的是运行时覆盖策略，不等于数据库字段迁移已完成**。  
  证据：`docs/update_log.md`、`docs/file_todo_list.md`、`docs/run_todo_list.md`
- 因此当前是“主链可先跑”，不是“数据库设计和持久化已经完全闭环”。  

### 4.5 检测正确性还没有完成闭环验证

- `main.py` 文档字符串已经明确写出：
  - 离线模式不包含跨帧 `track` 跟踪
  - 不包含完整时序约束
  - 不能直接等价为在线告警结果  
  证据：`inspection-flask/main.py`
- `docs/update_log.md`、`docs/file_todo_list.md` 也多次强调：**当前不能宣称“检测正确性已闭环验证”**。  
  证据：`docs/update_log.md`、`docs/file_todo_list.md`
- 当前更准确的说法应该是：
  - 主链已经形成
  - 规则口径已经统一
  - 但真实效果仍依赖权重、ROI、真实视频流和现场样例继续验证  

### 4.6 Python 版本要求存在不一致风险

- 仓库约定默认 Python 环境是 `yolo_code`，版本为 `3.9.25`。  
  证据：根目录 `AGENTS.md`、`docs/check_log.md`
- 但 `inspection-flask/pyproject.toml` 当前写的是 `requires-python = ">=3.10"`。  
  证据：`inspection-flask/pyproject.toml`
- 这会带来一个工程风险：**代码静态上能在 3.9 下编译，不代表依赖元数据安装链路与默认环境完全一致。**  

### 4.7 当前后台仍然是“最小骨架”，不是“完整后台系统”

- 当前已经有页面、路由、健康检查和部分相机管理能力。  
  证据：`inspection-flask/applications/view/system/hk_camera.py`、`inspection-flask/templates/system/hk_camera/main.html`
- 但文档明确说明：当前阶段是**最小后台兼容**，不是旧项目级别的完整后台恢复。  
  证据：`docs/run_todo_list.md`、`docs/update_log.md`
- 当前 `AUTH_ENABLED=False` 的默认策略，本质上是为了“先跑起来”，不代表完整权限体系已经恢复。  
  证据：`inspection-flask/settings.py`、`inspection-flask/applications/common/user_auth.py`

### 4.8 自动化验证仍然偏弱

- 目前可见的验证主要还是：
  - `compileall`
  - `main.py check`
  - 文档化静态复核
  - 局部命令自检  
  证据：`docs/check_log.md`
- 当前仓库内**未见标准化 `tests` 目录或系统化回归测试体系**，说明后续联调和回归仍需要较多人工验证。  

## 5. 建议明天汇报时的推荐表述

可以用下面这段作为主口径：

> 目前 `inspection-flask` 已经不是“没有主链的半成品”，而是已经完成了 YOLOv8 工服检测主链、时序违规判定、证据保存、离线验证工具和最小 Flask 后台骨架的阶段性版本。  
> 现在的核心问题已经不再是“代码有没有写出来”，而是“运行闭环、资源闭环和正确性闭环还差最后几步”。  
> 因此当前阶段最准确的定位是：**主链已成形，工程闭环仍在收口**。

## 6. 下一步最值得优先推进的事项

### P1：补齐运行资源

- 在默认环境中补齐 Flask 服务依赖  
  依据：`docs/check_log.md`、`docs/run_todo_list.md`
- 补齐 `inspection-flask/weights` 下两套 YOLOv8 权重  
  依据：`inspection-flask/settings.py`、`docs/check_log.md`

### P2：打通单摄像头真实在线闭环

- 先用一个真实可读输入源完成最小在线闭环：
  - 本地视频
  - `stream_url`
  - 或真实 RTSP  
  依据：`inspection-flask/applications/common/hk_recorder_threading.py`
- 若要进入更真实生产场景，再补海康原生 SDK 闭环  
  依据：`docs/run_todo_list.md`

### P3：补字段与持久化闭环

- 将当前运行时覆盖的 `roi / frame_path / stream_url` 进一步补齐为正式数据库字段与迁移方案  
  依据：`docs/file_todo_list.md`、`docs/run_todo_list.md`

### P4：做正确性验证而不是只看“能不能跑”

- 结合真实权重、真实 ROI、真实视频流和样例复核，验证：
  - 误报
  - 漏报
  - 告警抑制是否合理
  - 时序触发阈值是否合适  
  依据：`inspection-flask/main.py`、`docs/check_log.md`

### P5：统一工程环境约束

- 处理好 `pyproject.toml` 的 Python 版本要求与默认 `yolo_code` 环境之间的差异，避免后续安装链路继续产生歧义。  
  依据：`inspection-flask/pyproject.toml`、根目录 `AGENTS.md`

## 7. 最终结论

如果用一句最稳妥的话总结当前项目状态：

**项目核心代码主链已经基本完成，但运行资源、工程环境和效果验证还没有完全闭环。**

换句话说：

- 可以说：**“主链已经做出来了，项目已进入收口阶段。”**
- 不建议说：**“系统已经完全跑通并且效果已经验证完成。”**
